<h2 id="프로그램과-프로세스"><a class="heading-link" aria-hidden="true" href="#프로그램과-프로세스">프로그램과 프로세스</a></h2>
<h3 id="프로그램-program"><a class="heading-link" aria-hidden="true" href="#프로그램-program">프로그램 (Program)</a></h3>
<p>프로그램은 윈도우의 <code>*.exe</code> 파일이나 Mac의 <code>*.dmg</code> 파일과 같은 <strong>컴퓨터에서 실행할 수 있는 파일</strong>을 의미합니다. 단, 아직 <strong>파일을 실행하지 않은 상태</strong>이기 때문에 정적 프로그램(Static Program)이라고 부릅니다.</p>
<ul>
<li>파일이 저장 장치에 있지만 메모리에는 올라가 있지 않은 정적인 상태</li>
<li>쉽게 말해 그냥 <strong>코드 덩어리</strong></li>
</ul>
<h3 id="프로세스-process"><a class="heading-link" aria-hidden="true" href="#프로세스-process">프로세스 (Process)</a></h3>
<p>프로세스는 프로그램을 실행시켜 정적인 프로그램이 동적으로 변하여 <strong>프로그램이 돌아가고 있는 상태</strong>를 말합니다. 즉, 컴퓨터에서 <strong>작업 중인 프로그램</strong>을 의미합니다.</p>
<ul>
<li>메모리에 적재되고 CPU 자원을 할당받아 프로그램이 실행되고 있는 상태</li>
<li>그 코드 덩어리를 실행한 것</li>
</ul>
<p>모든 프로그램은 운영체제가 실행되기 위한 메모리 공간을 할당해 줘야 실행될 수 있습니다. 프로그램을 실행하는 순간 파일은 컴퓨터 메모리에 올라가게 되고, 운영체제로부터 시스템 자원(CPU)을 할당받아 프로그램 코드를 실행시켜 우리가 서비스를 이용할 수 있게 됩니다.</p>
<h2 id="프로세스의-자원-구조"><a class="heading-link" aria-hidden="true" href="#프로세스의-자원-구조">프로세스의 자원 구조</a></h2>
<p>프로그램이 실행되어 프로세스가 만들어지면 운영체제로부터 메모리 공간을 할당받게 됩니다. 이 메모리 공간은 다음과 같은 4가지 영역으로 구성됩니다:</p>
<h3 id="메모리-영역-구성"><a class="heading-link" aria-hidden="true" href="#메모리-영역-구성">메모리 영역 구성</a></h3>
<pre><code>높은 주소
┌─────────────────┐
│     Stack       │ ← 동적 영역 (함수 호출 시 증가/감소)
│        ↓        │
│                 │
│        ↑        │
│      Heap       │ ← 동적 영역 (동적 할당 시 증가/감소)
│                 │
├─────────────────┤
│     Data        │ ← 정적 영역 (프로그램 시작 시 크기 결정)
│   (.data,       │
│   .rodata,      │
│   .bss)         │
├─────────────────┤
│     Code        │ ← 정적 영역 (프로그램 시작 시 크기 결정)
│    (Text)       │
└─────────────────┘
낮은 주소
</code></pre>
<h3 id="1-코드-영역-code--text"><a class="heading-link" aria-hidden="true" href="#1-코드-영역-code--text">1. 코드 영역 (Code / Text)</a></h3>
<ul>
<li><strong>역할</strong>: 프로그래머가 작성한 프로그램 함수들의 코드가 CPU가 해석 가능한 기계어 형태로 저장되어 있는 영역</li>
<li><strong>특징</strong>:
<ul>
<li>읽기 전용(Read-Only) 영역</li>
<li>프로그램 시작 시 크기가 결정되는 정적 영역</li>
<li>여러 프로세스가 같은 프로그램을 실행하면 코드 영역은 공유될 수 있음</li>
</ul>
</li>
</ul>
<h3 id="2-데이터-영역-data"><a class="heading-link" aria-hidden="true" href="#2-데이터-영역-data">2. 데이터 영역 (Data)</a></h3>
<ul>
<li><strong>역할</strong>: 코드가 실행되면서 사용하는 전역 변수나 각종 데이터들이 모여있는 영역</li>
<li><strong>특징</strong>: 프로그램 시작 시 크기가 결정되는 정적 영역</li>
<li><strong>세부 영역</strong>:</li>
</ul>
<table>
<thead>
<tr>
<th>영역</th>
<th>설명</th>
<th>예시</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>.data</strong></td>
<td>전역 변수 또는 static 변수 등 프로그램이 사용하는 데이터</td>
<td><code>int global_var = 10;</code></td>
</tr>
<tr>
<td><strong>.bss</strong></td>
<td>초기값 없는 전역 변수, static 변수가 저장</td>
<td><code>int uninitialized_var;</code></td>
</tr>
<tr>
<td><strong>.rodata</strong></td>
<td>const 같은 상수 키워드 선언된 변수나 문자열 상수가 저장</td>
<td><code>const int MAX_SIZE = 100;</code></td>
</tr>
</tbody>
</table>
<h3 id="3-스택-영역-stack"><a class="heading-link" aria-hidden="true" href="#3-스택-영역-stack">3. 스택 영역 (Stack)</a></h3>
<ul>
<li><strong>역할</strong>: 지역 변수와 같은 호출한 함수가 종료되면 되돌아올 임시적인 자료를 저장하는 독립적인 공간</li>
<li><strong>특징</strong>:
<ul>
<li>함수의 호출과 함께 할당되며, 함수의 호출이 완료되면 소멸</li>
<li>LIFO (Last In First Out) 구조</li>
<li>스택 포인터가 높은 주소에서 낮은 주소 방향으로 증가</li>
<li>스택 영역을 초과하면 <strong>Stack Overflow</strong> 에러 발생</li>
<li>프로세스가 실행되는 동안 크기가 늘어났다 줄어들기도 하는 동적 영역</li>
</ul>
</li>
</ul>
<p><strong>스택에 저장되는 것들:</strong></p>
<ul>
<li>지역 변수</li>
<li>함수 매개변수</li>
<li>함수의 반환 주소</li>
<li>함수 호출 정보</li>
</ul>
<h3 id="4-힙-영역-heap"><a class="heading-link" aria-hidden="true" href="#4-힙-영역-heap">4. 힙 영역 (Heap)</a></h3>
<ul>
<li><strong>역할</strong>: 생성자, 인스턴스와 같은 동적으로 할당되는 데이터들을 위해 존재하는 공간</li>
<li><strong>특징</strong>:
<ul>
<li>사용자에 의해 메모리 공간이 동적으로 할당되고 해제됨</li>
<li>프로세스가 실행되는 동안 크기가 늘어났다 줄어들기도 하는 동적 영역</li>
<li>스택과 반대로 낮은 주소에서 높은 주소 방향으로 증가</li>
<li>프로그래머가 직접 관리해야 함 (메모리 누수 주의)</li>
</ul>
</li>
</ul>
<p><strong>힙에 저장되는 것들:</strong></p>
<ul>
<li><code>malloc()</code>, <code>new</code> 등으로 동적 할당된 메모리</li>
<li>객체 인스턴스</li>
<li>동적 배열</li>
</ul>
<h3 id="정적-영역-vs-동적-영역"><a class="heading-link" aria-hidden="true" href="#정적-영역-vs-동적-영역">정적 영역 vs 동적 영역</a></h3>
<p>위의 그림에서 Stack과 Heap 영역이 위아래로 화살표가 표시된 것을 볼 수 있는데, 이는 다음과 같은 의미입니다:</p>
<ul>
<li>
<p><strong>정적 영역 (Code, Data)</strong>:</p>
<ul>
<li>프로그램이 시작될 때 크기가 결정됨</li>
<li>프로그램 실행 중 크기 변경 불가</li>
<li>컴파일 타임에 메모리 크기 결정</li>
</ul>
</li>
<li>
<p><strong>동적 영역 (Stack, Heap)</strong>:</p>
<ul>
<li>프로세스가 실행되는 동안 크기가 늘어났다 줄어들기도 함</li>
<li>런타임에 메모리 크기 결정</li>
<li>Stack은 함수 호출에 따라, Heap은 동적 할당에 따라 크기 변경</li>
</ul>
</li>
</ul>
<h3 id="여러-프로세스의-메모리-구조"><a class="heading-link" aria-hidden="true" href="#여러-프로세스의-메모리-구조">여러 프로세스의 메모리 구조</a></h3>
<p>프로그램이 여러 개 실행된다면 메모리에 프로세스들이 담길 주소 공간이 생성되게 되고, 각 프로세스마다 독립적인 Code, Data, Stack, Heap 공간이 만들어지게 됩니다.</p>
<pre><code>메모리
┌─────────────────────────┐
│   프로세스 A              │
│   ┌─────────────────┐   │
│   │     Stack       │   │
│   │      Heap       │   │
│   │     Data        │   │
│   │     Code        │   │
│   └─────────────────┘   │
├─────────────────────────┤
│   프로세스 B              │
│   ┌─────────────────┐   │
│   │     Stack       │   │
│   │      Heap       │   │
│   │     Data        │   │
│   │     Code        │   │
│   └─────────────────┘   │
└─────────────────────────┘
</code></pre>
<p>각 프로세스는 독립적인 메모리 공간을 가지므로, 한 프로세스의 메모리에 다른 프로세스가 직접 접근할 수 없습니다. 이는 프로세스 간 안정성과 보안을 보장합니다.</p>
<h2 id="스레드-thread"><a class="heading-link" aria-hidden="true" href="#스레드-thread">스레드 (Thread)</a></h2>
<h3 id="프로세스의-한계"><a class="heading-link" aria-hidden="true" href="#프로세스의-한계">프로세스의 한계</a></h3>
<p>과거에는 프로그램을 실행할 때 프로세스 하나만을 사용했습니다. 하지만 기술이 발전됨에 따라 프로그램이 복잡해지고 다채로워짐으로써 프로세스 작업 하나만을 사용해서 프로그램을 실행하기에는 한계가 있었습니다.</p>
<p>동일한 프로그램을 여러 개의 프로세스로 만들게 되면, 그만큼 메모리를 차지하고 CPU에서 할당받는 자원이 중복되게 됩니다. 스레드(Thread)는 이러한 프로세스 특성의 한계를 해결하기 위해 탄생했습니다.</p>
<h3 id="스레드의-개념"><a class="heading-link" aria-hidden="true" href="#스레드의-개념">스레드의 개념</a></h3>
<p>스레드란, 하나의 <strong>프로세스 내에서</strong> <strong>동시에 진행되는 작업 갈래, 흐름의 단위</strong>를 말합니다.</p>
<p>예를 들어, 웹 브라우저에서 파일을 다운로드하면서 동시에 다른 웹페이지를 탐색할 수 있는 것은 하나의 프로세스 내에서 여러 스레드가 동시에 작업을 수행하기 때문입니다.</p>
<h2 id="프로세스-vs-스레드-비교"><a class="heading-link" aria-hidden="true" href="#프로세스-vs-스레드-비교">프로세스 vs 스레드 비교</a></h2>
<h3 id="자원-공유"><a class="heading-link" aria-hidden="true" href="#자원-공유">자원 공유</a></h3>
<table>
<thead>
<tr>
<th>구분</th>

<h2 id="멀티프로세스-vs-멀티스레드"><a class="heading-link" aria-hidden="true" href="#멀티프로세스-vs-멀티스레드">멀티프로세스 vs 멀티스레드</a></h2>
<h3 id="멀티프로세스-multi-process"><a class="heading-link" aria-hidden="true" href="#멀티프로세스-multi-process">멀티프로세스 (Multi-Process)</a></h3>
<ul>
<li>여러 개의 프로세스를 동시에 실행</li>
<li>각 프로세스는 독립적인 메모리 공간을 가짐</li>
<li>프로세스 간 통신이 어렵고 비용이 높음</li>
<li>하나의 프로세스가 죽어도 다른 프로세스에 영향 없음</li>
</ul>
<p><strong>장점:</strong></p>
<ul>
<li>프로세스 간 독립성으로 안정성 높음</li>
<li>하나의 프로세스 오류가 다른 프로세스에 영향 없음</li>
</ul>
<p><strong>단점:</strong></p>
<ul>
<li>메모리 사용량이 많음</li>
<li>컨텍스트 스위칭 비용이 높음</li>
<li>프로세스 간 통신이 복잡함</li>
</ul>
<h3 id="멀티스레드-multi-thread"><a class="heading-link" aria-hidden="true" href="#멀티스레드-multi-thread">멀티스레드 (Multi-Thread)</a></h3>
<ul>
<li>하나의 프로세스 내에서 여러 스레드를 동시에 실행</li>
<li>스레드 간 메모리 공간을 공유</li>
<li>스레드 간 통신이 쉽고 비용이 낮음</li>
<li>하나의 스레드 오류가 전체 프로세스에 영향 줄 수 있음</li>
</ul>
<p><strong>장점:</strong></p>
<ul>
<li>메모리 사용량이 적음</li>
<li>컨텍스트 스위칭이 빠름</li>
<li>스레드 간 통신이 쉬움</li>
<li>자원 공유가 효율적</li>
</ul>
<p><strong>단점:</strong></p>
<ul>
<li>동기화 문제 발생 가능 (Race Condition, Deadlock)</li>
<li>하나의 스레드 오류가 전체 프로세스에 영향</li>
<li>디버깅이 어려움</li>
</ul>
<h2 id="실제-사용-예시"><a class="heading-link" aria-hidden="true" href="#실제-사용-예시">실제 사용 예시</a></h2>
<h3 id="멀티프로세스-사용-사례"><a class="heading-link" aria-hidden="true" href="#멀티프로세스-사용-사례">멀티프로세스 사용 사례</a></h3>
<ul>
<li>웹 브라우저의 탭 (각 탭이 독립적인 프로세스)</li>
<li>운영체제의 여러 프로그램 동시 실행</li>
<li>서버의 여러 워커 프로세스</li>
</ul>
<h3 id="멀티스레드-사용-사례"><a class="heading-link" aria-hidden="true" href="#멀티스레드-사용-사례">멀티스레드 사용 사례</a></h3>
<ul>
<li>웹 브라우저 내에서 파일 다운로드와 웹페이지 탐색 동시 수행</li>
<li>워드 프로세서에서 타이핑과 자동 저장 동시 수행</li>
<li>게임에서 렌더링과 물리 엔진 동시 처리</li>
<li>웹 서버의 요청 처리 (각 요청을 스레드로 처리)</li>
</ul>

1f:T8f4f,<h2 id="엔디안이란"><a class="heading-link" aria-hidden="true" href="#엔디안이란">엔디안이란?</a></h2>
<p><strong>엔디안</strong>은 메모리에 저장된 <strong>멀티바이트 데이터의 바이트 순서</strong>(Byte Order)를 나타내는 개념이다. 같은 데이터를 저장할 때 바이트를 어떤 순서로 배치하느냐에 따라 해석이 달라진다.</p>
<h2 id="엔디안이-적용되는-데이터-타입"><a class="heading-link" aria-hidden="true" href="#엔디안이-적용되는-데이터-타입">엔디안이 적용되는 데이터 타입</a></h2>
<h3 id="적용되는-데이터-타입"><a class="heading-link" aria-hidden="true" href="#적용되는-데이터-타입">적용되는 데이터 타입</a></h3>
<p>엔디안은 <strong>1바이트보다 큰 데이터 타입</strong>에만 적용된다:</p>

<h3 id="핵심-원리"><a class="heading-link" aria-hidden="true" href="#핵심-원리">핵심 원리</a></h3>
<ul>
<li><strong>1바이트 데이터</strong>: 엔디안 문제가 없다. 바이트가 하나뿐이므로 순서가 의미가 없다.</li>
<li><strong>멀티바이트 데이터</strong>: 2바이트 이상의 데이터는 바이트 순서에 따라 해석이 달라진다.</li>
</ul>
<h2 id="엔디안의-종류"><a class="heading-link" aria-hidden="true" href="#엔디안의-종류">엔디안의 종류</a></h2>
<h3 id="big-endian-빅-엔디안"><a class="heading-link" aria-hidden="true" href="#big-endian-빅-엔디안">Big Endian (빅 엔디안)</a></h3>
<p><strong>바이트를 낮은 주소부터 저장</strong>하는 방식이다.</p>
<pre><code>메모리 주소:  [0x1000] [0x1001] [0x1002] [0x1003]
데이터:      [  0x12] [  0x34] [  0x56] [  0x78]
            ↑ 여기서 시작
</code></pre>
<ul>
<li><strong>사람이 읽기 쉬운 순서</strong>: 왼쪽에서 오른쪽으로 읽으면 자연스럽다.</li>
<li><strong>사용 예시</strong>: 네트워크 프로토콜(IP 주소 등), 일부 프로세서(ARM의 일부 모드, PowerPC 등)</li>
</ul>
<h3 id="little-endian-리틀-엔디안"><a class="heading-link" aria-hidden="true" href="#little-endian-리틀-엔디안">Little Endian (리틀 엔디안)</a></h3>
<p><strong>바이트를 높은 주소부터 저장</strong>하는 방식이다.</p>
<pre><code>메모리 주소:  [0x1000] [0x1001] [0x1002] [0x1003]
데이터:      [  0x78] [  0x56] [  0x34] [  0x12]
                                       ↑ 여기서 시작
</code></pre>
<ul>
<li><strong>메모리 효율성</strong>: 산술연산은 하위 바이트에서 시작하기 때문에 리틀 엔디안 방식은 cpu 연산에 유리하다.</li>
<li><strong>사용 예시</strong>: x86/x64 아키텍처, 대부분의 개인용 컴퓨터</li>
</ul>
<h2 id="뒤집는-단위와-범위"><a class="heading-link" aria-hidden="true" href="#뒤집는-단위와-범위">뒤집는 단위와 범위</a></h2>
<h3 id="뒤집는-단위-바이트byte-단위"><a class="heading-link" aria-hidden="true" href="#뒤집는-단위-바이트byte-단위">뒤집는 단위: <strong>바이트(Byte) 단위</strong></a></h3>
<p>엔디안은 <strong>바이트 단위로 뒤집힌다</strong>. 비트 단위가 아니다.</p>
<h4 id="예시-0x12345678-4바이트-정수"><a class="heading-link" aria-hidden="true" href="#예시-0x12345678-4바이트-정수">예시: 0x12345678 (4바이트 정수)</a></h4>
<p><strong>Big Endian:</strong></p>
<pre><code>메모리 주소:  0x1000  0x1001  0x1002  0x1003
바이트 값:    0x12    0x34    0x56    0x78
</code></pre>
<p><strong>Little Endian:</strong></p>

<h3 id="2-파일-포맷"><a class="heading-link" aria-hidden="true" href="#2-파일-포맷">2. 파일 포맷</a></h3>
<p>바이너리 파일 포맷은 특정 엔디안을 가정한다. 예를 들어:</p>
<ul>
<li><strong>PNG</strong>: Big Endian</li>
<li><strong>GIF</strong>: Little Endian</li>
<li><strong>JPEG</strong>: Big Endian</li>
</ul>
<h3 id="3-프로세서-간-데이터-공유"><a class="heading-link" aria-hidden="true" href="#3-프로세서-간-데이터-공유">3. 프로세서 간 데이터 공유</a></h3>
<p>서로 다른 엔디안을 가진 프로세서 간 데이터를 공유할 때 변환이 필요하다.</p>
<h3 id="4-디버깅"><a class="heading-link" aria-hidden="true" href="#4-디버깅">4. 디버깅</a></h3>
<p>메모리 덤프를 읽을 때 엔디안을 고려해야 올바르게 해석할 수 있다.</p>
