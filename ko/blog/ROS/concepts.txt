1c:T1b9d,<h2 id="nodes-노드"><a class="heading-link" aria-hidden="true" href="#nodes-노드">Nodes (노드)</a></h2>
<p><strong>노드</strong>는 ROS 2 그래프의 <strong>참여자</strong>로, <strong>클라이언트 라이브러리</strong>를 통해 다른 노드와 통신한다.</p>
<ul>
<li><strong>위치</strong>: 같은 프로세스, 다른 프로세스, 다른 기기에서 실행되는 노드와 모두 통신할 수 있다.</li>
<li><strong>역할</strong>: ROS 그래프에서 <strong>계산의 단위</strong>이며, 각 노드는 <strong>하나의 논리적 일</strong>을 담당하는 것이 좋다.</li>
<li><strong>통신 방식</strong>
<ul>
<li><strong>토픽</strong>: 명명된 토픽에 <strong>발행(publish)</strong> 하거나 <strong>구독(subscribe)</strong> 해서 데이터를 주고받는다.</li>
<li><strong>서비스</strong>: <strong>서비스 클라이언트</strong>로 다른 노드에 연산을 요청하거나, <strong>서비스 서버</strong>로 다른 노드에 기능을 제공한다.</li>
<li><strong>액션</strong>: 장시간 작업은 <strong>액션 클라이언트</strong>로 다른 노드에 맡기거나, <strong>액션 서버</strong>로 다른 노드에 기능을 제공한다.</li>
</ul>
</li>
<li><strong>파라미터</strong>: 실행 중 동작을 바꾸기 위해 <strong>구성 가능한 파라미터</strong>를 노출할 수 있다.</li>
<li><strong>조합</strong>: 한 노드가 발행자·구독자·서비스 서버/클라이언트·액션 서버/클라이언트를 <strong>동시에</strong> 가지는 경우가 많다.</li>
</ul>
<p>노드 간 <strong>연결</strong>은 <strong>분산 디스커버리(distributed discovery)</strong> 과정을 통해 이루어진다.</p>
<h2 id="discovery-디스커버리"><a class="heading-link" aria-hidden="true" href="#discovery-디스커버리">Discovery (디스커버리)</a></h2>
<p>노드 간 <strong>디스커버리</strong>는 ROS 2 <strong>미들웨어</strong>에 의해 <strong>자동</strong>으로 이루어진다.</p>
<ol>
<li><strong>시작 시</strong>: 노드가 시작되면, <strong>같은 ROS 도메인</strong>에 있는 다른 노드들에게 자신의 존재를 <strong>광고</strong>한다. (ROS 도메인은 환경 변수 <code>ROS_DOMAIN_ID</code>로 설정한다.) 다른 노드들은 이 광고에 자신에 대한 정보를 담아 응답하고, 그에 따라 <strong>연결이 맺어져</strong> 통신이 가능해진다.</li>
<li><strong>주기 광고</strong>: 노드는 <strong>주기적으로</strong> 자신의 존재를 광고한다. 그래서 초기 디스커버리 이후에 나중에 켜진 노드와도 연결할 수 있다.</li>
<li><strong>종료 시</strong>: 노드가 <strong>오프라인</strong>이 될 때도 다른 노드들에게 알린다.</li>
</ol>
<p><strong>연결 조건</strong>: 노드끼리 <strong>Quality of Service(QoS) 설정이 호환</strong>될 때만 실제 연결이 수립된다.</p>
<p>예: C++ talker를 한 터미널에서, Python listener를 다른 터미널에서 실행하면, 같은 토픽 이름으로 <strong>자동으로 서로를 찾아</strong> 메시지를 주고받기 시작한다.</p>

<h2 id="messages"><a class="heading-link" aria-hidden="true" href="#messages">Messages</a></h2>
<p><strong>메시지</strong>는 한 노드가 다른 노드에게 <strong>응답을 기대하지 않고</strong> 데이터를 보낼 때 쓰인다. 예: 센서에서 온도 데이터를 읽어 <code>Temperature</code> 메시지로 토픽에 발행하면, 다른 노드가 구독해 그 메시지를 받는다.</p>
<p>메시지는 패키지의 <code>msg/</code> 디렉터리 안 <code>.msg</code> 파일로 정의한다. <strong>필드(fields)</strong> 와 <strong>상수(constants)</strong> 로 구성된다.</p>
<h3 id="fields-필드"><a class="heading-link" aria-hidden="true" href="#fields-필드">Fields (필드)</a></h3>
<p>각 필드는 <strong>타입</strong>과 <strong>이름</strong>을 공백으로 구분해 한 줄에 쓴다.</p>
<pre><code>fieldtype1 fieldname1
fieldtype2 fieldname2
</code></pre>
<p>예: <code>int32 my_int</code>, <code>string my_string</code></p>
<ul>
<li><strong>필드 타입</strong>: (1) <strong>기본 타입</strong> — 아래 표. (2) <strong>다른 메시지 타입</strong> — 예: <code>geometry_msgs/PoseStamped</code>.</li>
<li><strong>필드 이름</strong>: 소문자·숫자·밑줄(<code>_</code>)만 사용. 알파벳으로 시작하고, 밑줄로 끝나거나 연속 밑줄(<code>__</code>)은 안 됨.</li>
<li><strong>기본값</strong>: 세 번째 요소로 지정 가능. 예: <code>uint8 x 42</code>, <code>string full_name "John Doe</code>. 문자열/문자열 배열·복합 타입에는 현재 기본값 미지원.</li>
</ul>

<p>기본 타입은 모두 <strong>배열</strong>로 쓸 수 있다: <code>int32[]</code>(동적 무제한), <code>int32[5]</code>(고정 5개), <code>int32[&#x3C;=5]</code>(최대 5개), <code>string&#x3C;=10</code>(최대 10자 문자열) 등.</p>
<h3 id="constants-상수"><a class="heading-link" aria-hidden="true" href="#constants-상수">Constants (상수)</a></h3>
<p>필드와 비슷하지만 <strong>대입 후 변경 불가</strong>하다. <code>=</code> 로 값을 넣고, <strong>이름은 대문자</strong>로 쓴다.</p>
<pre><code>int32 X=123
string FOO="foo"
</code></pre>
<h2 id="services"><a class="heading-link" aria-hidden="true" href="#services">Services</a></h2>
<p>서비스는 <strong>요청/응답</strong> 통신이다. 클라이언트가 서버에게 짧은 연산을 요청하고, 서버가 결과를 반환한다.</p>
<p>서비스는 패키지의 <code>srv/</code> 디렉터리 안 <code>.srv</code> 파일로 정의한다. <strong>요청</strong>과 <strong>응답</strong> 메시지 선언을 <code>---</code> 로 구분한다. <code>---</code> 로 이어진 두 개의 <code>.msg</code> 내용이 곧 유효한 서비스 정의다.</p>
<p>간단한 예 (문자열 받아 문자열 반환):</p>
<pre><code>string str
---
string str
</code></pre>
<p>요청/응답 각각에 상수·필드·다른 패키지 메시지 타입을 쓸 수 있다. <strong>서비스 안에 다른 서비스를 넣을 수는 없다.</strong></p>
<h2 id="actions"><a class="heading-link" aria-hidden="true" href="#actions">Actions</a></h2>
<p>액션은 <strong>장시간 요청/응답</strong> 통신이다. 액션 클라이언트가 서버에게 작업을 요청하고, 서버가 진행 중 <strong>피드백</strong>을 보내며 결과를 반환하며, <strong>중단</strong>도 가능하다.</p>
<p>액션은 <strong>goal</strong> / <strong>result</strong> / <strong>feedback</strong> 세 부분을 <code>---</code> 로 구분해 정의한다. 각 부분은 0개 이상의 필드를 가질 수 있으며, 타입·필드 이름 규칙은 메시지와 동일하다. 액션 정의는 패키지의 <strong><code>action/</code></strong> 폴더에 <code>.action</code> 파일로 둔다</p>
<p>예 (피보나치 액션):</p>
<pre><code>int32 order
---
int32[] sequence
---
int32[] sequence
</code></pre>
<p>클라이언트는 <code>order</code>(몇 번째까지 계산할지)를 goal로 보내고, 서버는 최종 수열을 result로, 중간 수열을 feedback으로 보낸다.</p>

29:T1d22,<p>**토픽(Topic)**은 ROS 2의 세 가지 주요 통신 방식 중 하나다. <strong>연속적인 데이터 스트림</strong>—센서 데이터, 로봇 상태 등—에 사용한다.</p>
<p>ROS 2는 **강타입(strongly-typed)**이고 **익명(anonymous)**인 <strong>발행/구독(publish/subscribe)</strong> 시스템이다. 아래에서 이 특성을 정리한다.</p>
<h2 id="publishsubscribe"><a class="heading-link" aria-hidden="true" href="#publishsubscribe">Publish/Subscribe</a></h2>
<p>데이터를 **만드는 쪽(Publisher)**과 **받는 쪽(Subscriber)**가 있다. 서로 **토픽(topic)**이라는 공통 이름으로 연결된다. 퍼블리셔를 만들 때 토픽 이름을 지정하고, 같은 토픽 이름으로 구독하는 서브스크라이버와 자동으로 통신한다.</p>
<ul>
<li>한 토픽에 <strong>퍼블리셔 0개 이상, 서브스크라이버 0개 이상</strong> 존재할 수 있다.</li>
<li>어떤 퍼블리셔가 토픽에 데이터를 보내면 <strong>그 토픽을 구독하는 모든 서브스크라이버</strong>가 그 데이터를 받는다.</li>
<li>이 구조를 **버스(bus)**에 비유한다. 퍼블리셔와 서브스크라이버가 필요에 따라 붙었다 떨어졌다 해도 되고, 그래서 디버깅·관측이 자연스럽다. 예: <code>ros2 bag record</code>는 지정한 토픽에 <strong>새 서브스크라이버를 하나 더 만드는 것</strong>일 뿐이라, 다른 노드들의 데이터 흐름을 끊지 않는다.</li>
</ul>
<h2 id="anonymous-익명"><a class="heading-link" aria-hidden="true" href="#anonymous-익명">Anonymous (익명)</a></h2>
<p><strong>익명</strong>이란, 서브스크라이버가 데이터를 받을 때 <strong>어느 퍼블리셔가 보냈는지 보통은 모르고 알 필요도 없다</strong>는 뜻이다(필요하면 조회는 가능). 그래서 퍼블리셔나 서브스크라이버를 바꿔 끼워도 시스템 나머지에 영향을 주지 않는다.</p>
<h2 id="strongly-typed-강타입"><a class="heading-link" aria-hidden="true" href="#strongly-typed-강타입">Strongly-typed (강타입)</a></h2>
<p><strong>강타입</strong>은 두 가지 의미가 있다.</p>
<ol>
<li><strong>필드 타입이 정해져 있고 검사된다.</strong> 메시지에 <code>uint32 field1</code>, <code>string field2</code>가 있으면, <code>field1</code>은 항상 부호 없는 정수, <code>field2</code>는 항상 문자열로 다뤄진다.</li>
<li><strong>필드의 의미(시맨틱)가 정의되어 있다.</strong> 자동으로 강제되지는 않지만, ROS 표준 메시지들은 의미가 명시되어 있다. 예: IMU 메시지의 각속도는 3차원 벡터이고, 단위는 rad/s로 정해져 있다.</li>
</ol>

2a:T1ca8,<p>ROS 2에서 **서비스(Service)**는 **원격 프로시저 호출(RPC)**을 의미한다. 한 노드가 다른 노드에게 계산을 요청하고, 그 노드가 연산을 수행한 뒤 <strong>결과를 한 번 반환</strong>하는 구조다.</p>
<blockquote>
<p><strong>RPC(Remote Procedure Call)</strong><br>
네트워크로 연결된 다른 프로세스의 함수를 로컬 함수처럼 호출하는 기술. 클라이언트가 요청을 보내면 서버가 계산 후 응답을 돌려준다.</p>
</blockquote>
<p>서비스 메시지 정의는 **요청(request)**과 <strong>응답(response)</strong> 두 부분으로 나뉜다.</p>
<pre><code>uint32 a
uint32 b
---
uint32 sum
</code></pre>
<p>ROS 2에서는 <strong>클라이언트가 결과를 기다리는</strong> 방식이므로, 서비스는 <strong>빠르게 응답하는 작업</strong>에 쓴다. 오래 걸리거나 중간에 취소·선점이 필요하거나 진행 피드백이 필요한 작업에는 <strong>액션</strong>을 사용한다. 서비스를 쓰면 안 되는 경우 예: 긴 계산, 로봇 이동, 취소(preempt) 가능 작업, 상태 피드백이 필요한 작업.</p>
<p>서비스는 <strong>서비스 이름</strong>(토픽 이름과 비슷하지만 별도 네임스페이스)으로 식별된다. 서비스는 <strong>Service Server</strong>와 <strong>Service Client</strong> 두 부분으로 구성된다.</p>
<h2 id="service-server"><a class="heading-link" aria-hidden="true" href="#service-server">Service Server</a></h2>
<p><strong>Service Server</strong>는 원격 프로시저 <strong>요청을 받아 연산을 수행하고 결과를 반환하는</strong> 쪽이다. 위 예에서 서버는 <code>a</code>와 <code>b</code>를 받아 더한 뒤 <code>sum</code>을 반환한다.</p>
<blockquote>
<p>하나의 서비스 이름에는 <strong>Service Server는 하나만</strong> 있어야 한다. 같은 이름에 서버가 여러 개면 어떤 서버가 요청을 받을지 정의되지 않는다.</p>
</blockquote>
<h2 id="service-client"><a class="heading-link" aria-hidden="true" href="#service-client">Service Client</a></h2>
<p><strong>Service Client</strong>는 Service Server에게 <strong>연산을 요청하는</strong> 쪽이다. 위 예에서 클라이언트는 <code>a</code>, <code>b</code>를 담은 요청을 보내고, 서버가 <code>sum</code>을 계산해 돌려줄 때까지 대기한다. <strong>Service Client는 동일 서비스 이름에 여러 개</strong> 존재할 수 있다.</p>

2b:T1f72,<p>ROS 2에서 **액션(Action)**은 <strong>피드백과 취소·선점(preempt)이 가능한 장시간 원격 프로시저 호출</strong>이다. 예: 상위 상태 기계가 내비게이션에 “웨이포인트까지 이동해라”고 액션을 호출하면, 수 초~수 분 걸리는 동안 진행 상황 피드백을 받고, 필요 시 목표(goal)를 취소하거나 선점할 수 있다.</p>
<p>액션 메시지 정의는 <strong>요청(goal)</strong> / <strong>응답(result)</strong> / <strong>피드백(feedback)</strong> 세 부분으로 구성된다.</p>
<pre><code>int32 order        # goal: 요청
---
int32[] sequence   # result: 최종 결과
---
int32[] sequence   # feedback: 진행 중 피드백
</code></pre>
<p>장시간 작업에는 액션을, <strong>짧은 RPC</strong>에는 서비스를 사용한다. 액션은 연결 설정·모니터링 오버헤드가 있으므로 단시간 호출에는 서비스가 적합하다. 액션은 <strong>액션 이름</strong>(토픽 이름과 비슷하지만 별도 네임스페이스)으로 식별된다.</p>
<h2 id="action-server"><a class="heading-link" aria-hidden="true" href="#action-server">Action Server</a></h2>
<p><strong>Action Server</strong>는 원격 프로시저 요청(goal)을 <strong>받아 실행하는 쪽</strong>이다. 실행 중 피드백을 보내고, 취소·선점 요청에 반응해야 한다. 예: 피보나치 수열 계산 액션 — <code>order</code>를 받아 수열을 계산하며 중간 <code>sequence</code>를 피드백하고, 최종 결과를 <code>sequence</code>로 반환.</p>
<blockquote>
<p>하나의 액션 이름에는 <strong>Action Server는 하나만</strong> 있어야 한다. 같은 이름에 서버가 여러 개면 어떤 서버가 요청을 받을지 정의되지 않는다.</p>
</blockquote>
<h2 id="action-client"><a class="heading-link" aria-hidden="true" href="#action-client">Action Client</a></h2>
<p><strong>Action Client</strong>는 Action Server에게 <strong>작업(goal) 수행을 요청하는 쪽</strong>이다. 위 예에서 클라이언트는 <code>order</code>를 담은 goal을 보내고, 서버가 수열을 계산해 결과(result)와 피드백을 돌려줄 때까지 대기한다. <strong>Action Client는 동일 액션 이름에 여러 개</strong> 존재할 수 있다.</p>

<h2 id="parameters-background"><a class="heading-link" aria-hidden="true" href="#parameters-background">Parameters background</a></h2>
<h3 id="declaring-parameters-선언"><a class="heading-link" aria-hidden="true" href="#declaring-parameters-선언">Declaring parameters (선언)</a></h3>
<p>기본적으로 노드는 <strong>수명 동안 받아들일 모든 파라미터를 미리 선언</strong>해야 한다. 이렇게 하면 타입과 이름이 노드 시작 시점에 정해져 나중에 잘못된 설정 가능성이 줄어든다. 선언하지 않은 파라미터도 허용하려면 노드 생성 시 <code>allow_undeclared_parameters: true</code>로 설정할 수 있다.</p>
<h3 id="parameter-types-타입"><a class="heading-link" aria-hidden="true" href="#parameter-types-타입">Parameter types (타입)</a></h3>
<p>각 파라미터는 위 Overview에 나온 사전 정의 타입 중 하나를 가진다. 기본적으로 <strong>선언된 파라미터의 타입을 런타임에 바꾸려고 하면 실패</strong>한다(예: 정수 파라미터에 불리언 대입 방지). 여러 타입을 허용하려면 선언 시 <code>ParameterDescriptor</code>의 <code>dynamic_typing: true</code>를 사용한다.</p>
<h3 id="parameter-callbacks-콜백"><a class="heading-link" aria-hidden="true" href="#parameter-callbacks-콜백">Parameter callbacks (콜백)</a></h3>
<p>누군가 노드의 파라미터를 <strong>바꾸려고 할 때</strong>, 그 전·중·후에 끼어들 수 있는 **세 가지 훅(hook)**이 있다. 모두 선택 사항이다.</p>
<p><strong>흐름</strong>: (요청 들어옴) → <strong>pre set</strong> → <strong>set</strong> → (실제 반영) → <strong>post set</strong></p>
<ul>
<li><strong>pre set</strong> (<code>add_pre_set_parameters_callback</code>): “아직 반영되기 전”에 호출된다. “이번에 바뀌려는 값 목록”을 받아서 <strong>그 목록을 수정</strong>할 수 있다. 예: <code>max_speed</code>가 바뀔 때마다 <code>min_speed</code>도 같이 바꾸고 싶다면, 여기서 목록에 <code>min_speed</code> 변경을 추가하면 된다.</li>
<li><strong>set</strong> (<code>add_on_set_parameters_callback</code>): “이 값으로 바꿔도 될지” <strong>승인/거부</strong>만 하는 단계다. 받은 값을 검사해서 “허용” 또는 “거부”를 반환한다. 여기서는 <strong>다른 상태를 바리거나 부수 효과를 두면 안 된다</strong> — 여러 콜백이 순서대로 실행되기 때문에, 나중 콜백에서 거부하면 실제로는 반영이 안 되는데 이미 상태만 바뀌어서 어긋날 수 있다.</li>
<li><strong>post set</strong> (<code>add_post_set_parameters_callback</code>): <strong>실제로 파라미터가 반영된 뒤</strong>에 한 번 호출된다. “값이 바뀌었으니 이제 뭔가 반응해야 할 때” 쓰면 된다. 예: <code>max_speed</code>가 바뀌었으면 내부 제한치를 갱신하거나 로그를 남기는 등.</li>
</ul>

<h2 id="공식-지원-rclcpp--rclpy"><a class="heading-link" aria-hidden="true" href="#공식-지원-rclcpp--rclpy">공식 지원: rclcpp / rclpy</a></h2>
<p>C++용 <strong>rclcpp</strong>와 Python용 <strong>rclpy</strong>는 둘 다 <strong>공통 코어 rcl</strong>을 사용하는 공식 클라이언트 라이브러리다.</p>
<ul>
<li><strong>rclcpp</strong>: C++ 관용 인터페이스. 노드·퍼블리셔·서브스크라이버 등 ROS 클라이언트 기능 제공. <code>rcl</code>과 <code>rosidl</code> API 위에 구축되며, <code>rosidl_generator_cpp</code>로 생성된 C++ 메시지와 함께 사용. C++17 활용. <a href="https://docs.ros.org/en/jazzy/p/rclcpp/">API 문서</a>.</li>
<li><strong>rclpy</strong>: Python용. <code>rcl</code> C API 위에 구축, Python 관용 타입·패턴(list, context 등) 제공. 실행 모델은 <code>threading.Thread</code> 등으로 <code>rcl</code> API를 실행. 메시지는 Python 객체로 다루다가 <code>rcl</code> 계층으로 넘길 때 C 메시지로 변환(같은 프로세스 내 통신 시 변환 최소화). <a href="https://docs.ros.org/en/jazzy/p/rclpy/">API 문서</a>.</li>
</ul>
<h2 id="공통-코어-rcl"><a class="heading-link" aria-hidden="true" href="#공통-코어-rcl">공통 코어: rcl</a></h2>
<p>언어에 <strong>종속되지 않는</strong> 동작(파라미터, 네임스페이스 등)은 <strong>rcl</strong>(ROS Client Library)이라는 <strong>공통 C API</strong>에 구현되어 있다. 각 언어용 클라이언트 라이브러리는 이 rcl을 <strong>FFI로 래핑</strong>하는 형태라서, 라이브러리는 얇게 유지되고 동작이 <strong>언어 간에 일관</strong>된다. rcl 쪽 로직을 바꾸면 rcl을 쓰는 모든 클라이언트에 반영되며, 버그 수정도 한 곳에서 하면 된다.</p>
<p><strong>언어별로만 구현되는 부분</strong>(예: “spin” 함수의 스레딩 모델)은 각 클라이언트 라이브러리에서 따로 구현한다.</p>
<blockquote>
<p>**FFI(Foreign Function Interface)**란?
한 언어에서 <strong>다른 언어로 작성된 함수</strong>를 호출할 수 있게 하는 인터페이스다. 여기서는 C로 구현된 rcl을 Python·C++ 등에서 호출하는 데 쓰인다.</p>
</blockquote>
<h2 id="커뮤니티-유지-라이브러리"><a class="heading-link" aria-hidden="true" href="#커뮤니티-유지-라이브러리">커뮤니티 유지 라이브러리</a></h2>
<p>C++·Python 외에 커뮤니티에서 유지하는 클라이언트 라이브러리 예: <strong>Ada</strong>, <strong>C</strong>(rclc, micro-ROS), <strong>JVM/Android</strong>(Java, Android), <strong>.NET/C#</strong>, <strong>Node.js</strong>(rclnodejs), <strong>Rust</strong>(rclrs), <strong>Flutter/Dart</strong> 등. C#, Objective C/iOS, Zig 등은 구버전·미유지 상태로 문서에 명시되어 있다.</p>
<h2 id="ros-1과의-차이"><a class="heading-link" aria-hidden="true" href="#ros-1과의-차이">ROS 1과의 차이</a></h2>
<p>ROS 1에서는 각 클라이언트 라이브러리가 <strong>처음부터 따로</strong> 만들어져, 예를 들어 Python은 순수 Python으로만 구현되는 장점이 있지만, <strong>이름 규칙·동작이 라이브러리마다 다르고</strong>, 버그 수정을 여러 곳에 해야 하며, UDPROS처럼 <strong>한 라이브러리에만 있는 기능</strong>도 많다. ROS 2는 <strong>공통 코어(rcl)</strong> 를 두어 여러 언어용 클라이언트를 더 쉽게 만들고 동작을 맞추었다.</p>

<h2 id="도메인-id-선택"><a class="heading-link" aria-hidden="true" href="#도메인-id-선택">도메인 ID 선택</a></h2>
<p><strong>안전하게 쓰려면 도메인 ID는 0 이상 101 이하(포함)로 두는 것이 좋다.</strong> 아래는 그 이유(포트 계산, 플랫폼별 일시 포트 범위, 참가자 수 제약)를 정리한 것이다.</p>
<h2 id="도메인-id와-udp-포트"><a class="heading-link" aria-hidden="true" href="#도메인-id와-udp-포트">도메인 ID와 UDP 포트</a></h2>
<p>도메인 ID는 DDS가 <strong>디스커버리·통신용 UDP 포트</strong>를 계산하는 데 쓰인다. UDP 포트는 16비트 부호 없는 정수이므로 상한은 65535이고, 공식 문서의 계산식에 따르면 <strong>도메인 ID는 0~232</strong>까지 쓸 수 있다.</p>
<h2 id="플랫폼별-제약-일시-포트와의-충돌-방지"><a class="heading-link" aria-hidden="true" href="#플랫폼별-제약-일시-포트와의-충돌-방지">플랫폼별 제약 (일시 포트와의 충돌 방지)</a></h2>

<h2 id="참가자프로세스-수-제약"><a class="heading-link" aria-hidden="true" href="#참가자프로세스-수-제약">참가자(프로세스) 수 제약</a></h2>
<p>컴퓨터에서 실행되는 ROS 2 프로세스(ros2 run 으로 실행된 실행단위)마다 DDS <strong>참가자(participant)</strong> 하나가 만들어진다. 참가자 하나는 <strong>UDP 포트 2개</strong>를 사용한다. 따라서 <strong>한 컴퓨터에서 ROS 2 프로세스를 120개 넘게 띄우면</strong>, 사용하는 포트가 <strong>다른 도메인 ID용 포트</strong>나 <strong>일시(ephemeral) 포트</strong>와 겹칠 수 있다.</p>
<p>그 이유를 도메인 ID 1과 2로 보면 다음과 같다(<a href="https://docs.ros.org/en/jazzy/Concepts/Intermediate/About-Domain-ID.html">원문</a> Participant constraints).</p>
<ul>
<li>도메인 ID 1은 <strong>멀티캐스트</strong>에 포트 7650, 7651을 쓴다.</li>
<li>도메인 ID 2는 <strong>멀티캐스트</strong>에 포트 7900, 7901을 쓴다.</li>
<li>도메인 ID 1에서 **1번째 프로세스(0번째 참가자)**는 <strong>유니캐스트</strong>에 7660, 7661을 쓴다.</li>
<li>도메인 ID 1에서 **120번째 프로세스(119번째 참가자)**는 유니캐스트에 7898, 7899를 쓴다.</li>
<li>도메인 ID 1에서 **121번째 프로세스(120번째 참가자)**는 유니캐스트에 7900, 7901을 쓰게 되는데, 이 포트는 이미 도메인 ID 2의 멀티캐스트용으로 쓰이므로 <strong>겹친다</strong>.</li>
</ul>
<p>해당 컴퓨터가 <strong>한 번에 하나의 도메인 ID만</strong> 쓰고, 그 도메인 ID가 <strong>충분히 낮으면</strong> 120개를 넘는 ROS 2 프로세스를 띄워도 괜찮다.</p>
<p>도메인 ID를 <strong>플랫폼별 안전 범위의 상단</strong>(예: Linux에서 101, 232, macOS/Windows에서 166)에 두는 경우에는 한 가지를 더 고려해야 한다. 유니캐스트 포트가 큰 번호로 밀리기 때문에, <strong>120개보다 훨씬 적은</strong> 프로세스 수에서부터 <strong>일시(ephemeral) 포트</strong>와 겹치기 시작한다.</p>
<p>예: Linux에서 도메인 ID 101일 때</p>
<ul>
<li>0번째 ROS 2 프로세스: 32650, 32651, 32660, 32661</li>
<li>1번째 프로세스: 32650, 32651, 32662, 32663</li>
<li>53번째 프로세스: 32650, 32651, 32766, 32767</li>
<li><strong>54번째 프로세스</strong>: 32650, 32651, <strong>32768, 32769</strong> → Linux 일시 포트 범위(32768–60999)와 겹친다.</li>
</ul>
<p>따라서 Linux에서 도메인 ID 101을 쓸 때는 프로세스는 <strong>54개</strong>까지 두는 것이 좋다. Linux에서 도메인 ID 232는 포트 상한 65535 때문에 <strong>63개</strong> 정도가 한계이고, macOS/Windows에서 도메인 ID 166은 <strong>120개</strong>가 한계다.</p>

<blockquote>
<p><strong>DDS란?</strong><br>
<strong>Data Distribution Service</strong>의 약자로, OMG에서 정한 <strong>데이터 중심(data-centric) publish-subscribe</strong> 표준이다. 토픽·QoS·분산 디스커버리(중앙 브로커 없이 참가자끼리 서로 찾음)를 제공하며, 로봇·항공·의료 등 실시간·분산 시스템에서 널리 쓴다. "무엇을(토픽/타입)·어떤 품질로(QoS) 주고받을지"를 정의하는 <strong>동작·API 표준</strong>이라고 보면 된다.</p>
</blockquote>
<blockquote>
<p><strong>OMG란?</strong><br>
<strong>Object Management Group</strong>의 약자로, DDS·CORBA·UML 등 소프트웨어 표준을 정의·유지하는 <strong>국제 표준화 단체</strong>다. 1989년 설립되었고, 업계·학계·정부가 참여하는 컨소시엄 형태로 운영된다.</p>
</blockquote>
<blockquote>
<p><strong>RTPS란?</strong><br>
<strong>Real-Time Publish-Subscribe</strong>의 약자로, DDS가 <strong>네트워크 위에서 실제로 패킷을 주고받을 때 쓰는 프로토콜</strong>이다. 정식 명칭은 DDSI-RTPS(DDS Interoperability Wire Protocol)다. 즉 DDS가 "무엇을 어떻게 할지"를 정한다면, RTPS는 "그걸 와이어(케이블/네트워크) 위에서 어떻게 직렬화·전송할지"를 정한다. 서로 다른 벤더의 DDS 구현체도 RTPS를 따르면 <strong>상호 연동</strong>할 수 있다.</p>
</blockquote>
<blockquote>
<p>공식 개념 문서: <a href="https://docs.ros.org/en/jazzy/Concepts/Intermediate/About-Different-Middleware-Vendors.html">Different ROS 2 middleware vendors</a> (Jazzy)</p>
</blockquote>

<h2 id="여러-rmw-구현체-사용"><a class="heading-link" aria-hidden="true" href="#여러-rmw-구현체-사용">여러 RMW 구현체 사용</a></h2>
<p>현재 활성 배포판의 <strong>바이너리</strong>에는 Fast DDS, RTI Connext Pro, Eclipse Cyclone DDS, GurumNetworks GurumDDS 등 여러 RMW가 기본 포함된다. <strong>기본값은 Fast DDS</strong>이며, 추가 설치 없이 사용 가능하다(바이너리 패키지에 함께 배포됨). Cyclone DDS, Connext, GurumDDS는 추가 패키지 설치로 활성화할 수 있고, 기존 패키지를 교체하거나 다시 빌드할 필요는 없다.</p>
<p><strong>소스에서 빌드</strong>한 워크스페이스는, 해당 DDS/RTPS가 설치되어 있고 환경 변수가 설정되어 있으면 <strong>여러 RMW를 동시에 빌드·설치</strong>할 수 있다. 예: RTI Connext DDS용 RMW 코드가 워크스페이스에 있고 Connext Pro가 설치되어 있으면 그 RMW가 빌드된다.</p>
<p>서로 <strong>다른 RMW를 쓰는 노드</strong>끼리도 많은 경우 통신이 되지만, <strong>항상 보장되지는 않는다</strong>. 아래 조합은 지원되지 않는다.</p>
<ul>
<li><strong>Fast DDS ↔ Connext</strong>: Fast DDS가 publish한 <code>WString</code>을 Connext가 macOS에서 올바르게 수신하지 못함</li>
<li><strong>Connext ↔ Cyclone DDS</strong>: <code>WString</code>에 대한 pub/sub 미지원</li>
</ul>
<h2 id="기본-rmw-구현체"><a class="heading-link" aria-hidden="true" href="#기본-rmw-구현체">기본 RMW 구현체</a></h2>
<p>여러 RMW가 설치된 ROS 2 환경에서는 <strong>Fast DDS</strong>가 있으면 <strong>기본 RMW</strong>로 선택된다. Fast DDS RMW가 없으면, <strong>RMW 구현체 식별자(패키지 이름) 알파벳 순</strong>으로 첫 번째가 기본이 된다. 예: <code>rmw_cyclonedds_cpp</code>와 <code>rmw_connextdds</code>만 있으면 <code>rmw_connextdds</code>가 기본이고, <code>rmw_fastrtps_cpp</code>를 나중에 설치하면 그때부터 Fast DDS가 기본이 된다.</p>
<p>실행 시 사용할 RMW를 지정하는 방법은 <a href="https://docs.ros.org/en/jazzy/How-To-Guides/Working-with-multiple-ROS-2-middleware-implementations.html">가이드</a>를 참고하면 된다.</p>
