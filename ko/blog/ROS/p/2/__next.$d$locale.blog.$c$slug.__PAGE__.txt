5:T2ce8,<p>ROS 2의 기본 미들웨어인 <strong>DDS</strong>에서는 <strong>Domain ID</strong>로 논리 네트워크를 나눈다. <strong>같은 도메인 ID</strong>를 쓰는 노드끼리는 서로 발견하고 통신할 수 있고, <strong>다른 도메인 ID</strong>를 쓰는 노드끼리는 통신하지 않는다. 기본값은 <strong>0</strong>이다. 같은 물리 네트워크에서 여러 팀/그룹이 ROS 2를 돌릴 때는 그룹마다 서로 다른 도메인 ID를 두면 간섭을 피할 수 있다.</p>
<blockquote>
<p>공식 개념 문서: <a href="https://docs.ros.org/en/jazzy/Concepts/Intermediate/About-Domain-ID.html">The ROS_DOMAIN_ID</a> (Jazzy)</p>
</blockquote>
<h2 id="도메인-id-선택"><a class="heading-link" aria-hidden="true" href="#도메인-id-선택">도메인 ID 선택</a></h2>
<p><strong>안전하게 쓰려면 도메인 ID는 0 이상 101 이하(포함)로 두는 것이 좋다.</strong> 아래는 그 이유(포트 계산, 플랫폼별 일시 포트 범위, 참가자 수 제약)를 정리한 것이다.</p>
<h2 id="도메인-id와-udp-포트"><a class="heading-link" aria-hidden="true" href="#도메인-id와-udp-포트">도메인 ID와 UDP 포트</a></h2>
<p>도메인 ID는 DDS가 <strong>디스커버리·통신용 UDP 포트</strong>를 계산하는 데 쓰인다. UDP 포트는 16비트 부호 없는 정수이므로 상한은 65535이고, 공식 문서의 계산식에 따르면 <strong>도메인 ID는 0~232</strong>까지 쓸 수 있다.</p>
<h2 id="플랫폼별-제약-일시-포트와의-충돌-방지"><a class="heading-link" aria-hidden="true" href="#플랫폼별-제약-일시-포트와의-충돌-방지">플랫폼별 제약 (일시 포트와의 충돌 방지)</a></h2>
<p>OS의 <strong>일시(ephemeral) 포트 범위</strong>와 겹치지 않도록 도메인 ID 범위를 제한하는 것이 좋다.</p>

<p>Linux에서 <strong>모든 플랫폼과의 호환</strong>을 고려하면 <strong>0–101</strong>을 쓰는 것이 안전하다. Linux에서 일시 포트 범위는 <code>/proc/sys/net/ipv4/ip_local_port_range</code>로, macOS/Windows에서는 각각 sysctl·netsh로 변경 가능하다. 사용자 정의 범위를 쓰면 위 숫자는 달라질 수 있다.</p>
<h2 id="참가자프로세스-수-제약"><a class="heading-link" aria-hidden="true" href="#참가자프로세스-수-제약">참가자(프로세스) 수 제약</a></h2>
<p>컴퓨터에서 실행되는 ROS 2 프로세스(ros2 run 으로 실행된 실행단위)마다 DDS <strong>참가자(participant)</strong> 하나가 만들어진다. 참가자 하나는 <strong>UDP 포트 2개</strong>를 사용한다. 따라서 <strong>한 컴퓨터에서 ROS 2 프로세스를 120개 넘게 띄우면</strong>, 사용하는 포트가 <strong>다른 도메인 ID용 포트</strong>나 <strong>일시(ephemeral) 포트</strong>와 겹칠 수 있다.</p>
<p>그 이유를 도메인 ID 1과 2로 보면 다음과 같다(<a href="https://docs.ros.org/en/jazzy/Concepts/Intermediate/About-Domain-ID.html">원문</a> Participant constraints).</p>
<ul>
<li>도메인 ID 1은 <strong>멀티캐스트</strong>에 포트 7650, 7651을 쓴다.</li>
<li>도메인 ID 2는 <strong>멀티캐스트</strong>에 포트 7900, 7901을 쓴다.</li>
<li>도메인 ID 1에서 **1번째 프로세스(0번째 참가자)**는 <strong>유니캐스트</strong>에 7660, 7661을 쓴다.</li>
<li>도메인 ID 1에서 **120번째 프로세스(119번째 참가자)**는 유니캐스트에 7898, 7899를 쓴다.</li>
<li>도메인 ID 1에서 **121번째 프로세스(120번째 참가자)**는 유니캐스트에 7900, 7901을 쓰게 되는데, 이 포트는 이미 도메인 ID 2의 멀티캐스트용으로 쓰이므로 <strong>겹친다</strong>.</li>
</ul>
<p>해당 컴퓨터가 <strong>한 번에 하나의 도메인 ID만</strong> 쓰고, 그 도메인 ID가 <strong>충분히 낮으면</strong> 120개를 넘는 ROS 2 프로세스를 띄워도 괜찮다.</p>
<p>도메인 ID를 <strong>플랫폼별 안전 범위의 상단</strong>(예: Linux에서 101, 232, macOS/Windows에서 166)에 두는 경우에는 한 가지를 더 고려해야 한다. 유니캐스트 포트가 큰 번호로 밀리기 때문에, <strong>120개보다 훨씬 적은</strong> 프로세스 수에서부터 <strong>일시(ephemeral) 포트</strong>와 겹치기 시작한다.</p>
<p>예: Linux에서 도메인 ID 101일 때</p>
<ul>
<li>0번째 ROS 2 프로세스: 32650, 32651, 32660, 32661</li>
<li>1번째 프로세스: 32650, 32651, 32662, 32663</li>
<li>53번째 프로세스: 32650, 32651, 32766, 32767</li>
<li><strong>54번째 프로세스</strong>: 32650, 32651, <strong>32768, 32769</strong> → Linux 일시 포트 범위(32768–60999)와 겹친다.</li>
</ul>
<p>따라서 Linux에서 도메인 ID 101을 쓸 때는 프로세스는 <strong>54개</strong>까지 두는 것이 좋다. Linux에서 도메인 ID 232는 포트 상한 65535 때문에 <strong>63개</strong> 정도가 한계이고, macOS/Windows에서 도메인 ID 166은 <strong>120개</strong>가 한계다.</p>

12:T19f1,<p>ROS 2는 <strong>DDS/RTPS</strong>를 미들웨어로 사용하며, 여기서 <strong>디스커버리·직렬화·전송</strong>이 이뤄진다. DDS는 분산 디스커버리(ROS 1처럼 중앙이 아님), 전송 <strong>QoS</strong> 제어 등 로봇 시스템에 맞는 기능을 제공하는 산업 표준이고, RTI Connext, eProsima Fast DDS, Eclipse Cyclone DDS, GurumNetworks GurumDDS 등 여러 벤더가 구현체를 제공한다. <strong>RTPS</strong>(DDSI-RTPS)는 DDS가 네트워크로 통신할 때 쓰는 <strong>와이어 프로토콜</strong>이다.</p>
<p>ROS 2에서 특정 DDS/RTPS 구현체를 쓰려면, 그 구현체의 API·도구로 ROS 미들웨어 추상 인터페이스를 구현한 <strong>RMW(ROS Middleware interface)</strong> 패키지가 필요하다. RMW 패키지를 여러 개 지원하는 것은 “한 구현체에만 묶이지 않게” 하기 위함이며, 사용자가 프로젝트에 맞게 구현체를 바꿀 수 있게 하기 위해서다.</p>
<blockquote>
<p><strong>DDS란?</strong><br>
<strong>Data Distribution Service</strong>의 약자로, OMG에서 정한 <strong>데이터 중심(data-centric) publish-subscribe</strong> 표준이다. 토픽·QoS·분산 디스커버리(중앙 브로커 없이 참가자끼리 서로 찾음)를 제공하며, 로봇·항공·의료 등 실시간·분산 시스템에서 널리 쓴다. "무엇을(토픽/타입)·어떤 품질로(QoS) 주고받을지"를 정의하는 <strong>동작·API 표준</strong>이라고 보면 된다.</p>
</blockquote>
<blockquote>
<p><strong>OMG란?</strong><br>
<strong>Object Management Group</strong>의 약자로, DDS·CORBA·UML 등 소프트웨어 표준을 정의·유지하는 <strong>국제 표준화 단체</strong>다. 1989년 설립되었고, 업계·학계·정부가 참여하는 컨소시엄 형태로 운영된다.</p>
</blockquote>
<blockquote>
<p><strong>RTPS란?</strong><br>
<strong>Real-Time Publish-Subscribe</strong>의 약자로, DDS가 <strong>네트워크 위에서 실제로 패킷을 주고받을 때 쓰는 프로토콜</strong>이다. 정식 명칭은 DDSI-RTPS(DDS Interoperability Wire Protocol)다. 즉 DDS가 "무엇을 어떻게 할지"를 정한다면, RTPS는 "그걸 와이어(케이블/네트워크) 위에서 어떻게 직렬화·전송할지"를 정한다. 서로 다른 벤더의 DDS 구현체도 RTPS를 따르면 <strong>상호 연동</strong>할 수 있다.</p>
</blockquote>

<h2 id="여러-rmw-구현체-사용"><a class="heading-link" aria-hidden="true" href="#여러-rmw-구현체-사용">여러 RMW 구현체 사용</a></h2>
<p>현재 활성 배포판의 <strong>바이너리</strong>에는 Fast DDS, RTI Connext Pro, Eclipse Cyclone DDS, GurumNetworks GurumDDS 등 여러 RMW가 기본 포함된다. <strong>기본값은 Fast DDS</strong>이며, 추가 설치 없이 사용 가능하다(바이너리 패키지에 함께 배포됨). Cyclone DDS, Connext, GurumDDS는 추가 패키지 설치로 활성화할 수 있고, 기존 패키지를 교체하거나 다시 빌드할 필요는 없다.</p>
<p><strong>소스에서 빌드</strong>한 워크스페이스는, 해당 DDS/RTPS가 설치되어 있고 환경 변수가 설정되어 있으면 <strong>여러 RMW를 동시에 빌드·설치</strong>할 수 있다. 예: RTI Connext DDS용 RMW 코드가 워크스페이스에 있고 Connext Pro가 설치되어 있으면 그 RMW가 빌드된다.</p>
<p>서로 <strong>다른 RMW를 쓰는 노드</strong>끼리도 많은 경우 통신이 되지만, <strong>항상 보장되지는 않는다</strong>. 아래 조합은 지원되지 않는다.</p>
<ul>
<li><strong>Fast DDS ↔ Connext</strong>: Fast DDS가 publish한 <code>WString</code>을 Connext가 macOS에서 올바르게 수신하지 못함</li>
<li><strong>Connext ↔ Cyclone DDS</strong>: <code>WString</code>에 대한 pub/sub 미지원</li>
</ul>
<h2 id="기본-rmw-구현체"><a class="heading-link" aria-hidden="true" href="#기본-rmw-구현체">기본 RMW 구현체</a></h2>

Fast DDS RMW가 없으면, <strong>RMW 구현체 식별자(패키지 이름) 알파벳 순</strong>으로 첫 번째가 기본이 된다. 예: <code>rmw_cyclonedds_cpp</code>와 <code>rmw_connextdds</code>만 있으면 <code>rmw_connextdds</code>가 기본이고, <code>rmw_fastrtps_cpp</code>를 나중에 설치하면 그때부터 Fast DDS가 기본이 된다.</p>

<h2 id="심각도-레벨severity-level"><a class="heading-link" aria-hidden="true" href="#심각도-레벨severity-level">심각도 레벨(Severity level)</a></h2>
<p>로그 메시지에는 <strong>심각도</strong>가 붙는다: <code>DEBUG</code>, <code>INFO</code>, <code>WARN</code>, <code>ERROR</code>, <code>FATAL</code>(낮은 순).</p>

14:T8549,<p>ROS 2는 <strong>Quality of Service(QoS)</strong> 정책으로 노드 간 통신을 조정한다. 설정에 따라 TCP처럼 <strong>신뢰(reliable)</strong> 하게 할 수도, UDP처럼 <strong>최선 노력(best effort)</strong> 으로 할 수도 있고, 그 사이 다양한 조합이 가능하다. ROS 1이 주로 TCP에 의존한 것과 달리, ROS 2는 DDS 전송의 유연성을 활용해 손실이 있는 무선 환경에서는 best effort가, 실시간 시스템에서는 적절한 QoS 프로파일이 쓰인다.</p>
<p>여러 <strong>QoS 정책(policies)</strong> 이 모여 <strong>QoS 프로파일(profile)</strong> 을 이룬다. 시나리오에 맞는 정책 선택이 복잡하므로 ROS 2는 센서 데이터 등 <strong>자주 쓰는 용도별 기본 프로파일</strong>을 제공하고, 필요 시 개별 정책을 덮어쓸 수 있다. QoS 프로파일은 <strong>퍼블리셔·서브스크립션·서비스 서버·클라이언트</strong> 각각에 지정할 수 있다. 서로 다른 프로파일을 쓰면 <strong>호환되지 않아</strong> 메시지가 전달되지 않을 수 있다.</p>
<blockquote>
<p>공식 개념 문서: <a href="https://docs.ros.org/en/jazzy/Concepts/Intermediate/About-Quality-of-Service-Settings.html">Quality of Service settings</a> (Jazzy)</p>
</blockquote>
<h2 id="qos-정책policies"><a class="heading-link" aria-hidden="true" href="#qos-정책policies">QoS 정책(Policies)</a></h2>
<p>기본 QoS 프로파일에 포함되는 정책은 다음과 같다.</p>
<table>
<thead>
<tr>
<th>정책</th>
<th>옵션</th>
<th>설명</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>History</strong></td>
<td>Keep last / Keep all</td>
<td>Keep last: 최대 N개 샘플만 유지(depth로 N 설정). Keep all: 모두 유지(미들웨어 리소스 한도 내).</td>
</tr>
<tr>
<td><strong>Depth</strong></td>
<td>Queue size</td>
<td>History가 "keep last"일 때만 적용.</td>
</tr>
<tr>
<td><strong>Reliability</strong></td>
<td>Best effort / Reliable</td>
<td>Best effort: 전달 시도하나 손실 가능. Reliable: 전달 보장, 재시도 가능.</td>
</tr>
<tr>
<td><strong>Durability</strong></td>
<td>Transient local / Volatile</td>
<td>Transient local: 늦게 붙는 서브스크립션을 위해 퍼블리셔가 샘플 유지. Volatile: 유지 안 함.</td>
</tr>
<tr>
<td><strong>Deadline</strong></td>
<td>Duration</td>
<td>토픽에 연속 메시지가 나갈 때 <strong>기대하는 최대 간격</strong>.</td>
</tr>
<tr>
<td><strong>Lifespan</strong></td>
<td>Duration</td>
<td>발행 후 수신까지 이 시간을 넘기면 <strong>만료</strong>로 간주하고 조용히 버림.</td>
</tr>
<tr>
<td><strong>Liveliness</strong></td>
<td>Automatic / Manual by topic</td>
<td>Automatic: 노드의 퍼블리셔 중 하나만 메시지를 보내도 노드가 살아 있다고 간주. Manual by topic: 퍼블리셔 API로 수동으로 liveliness 선언.</td>
</tr>
<tr>
<td><strong>Lease Duration</strong></td>
<td>Duration</td>
<td>퍼블리셔가 이 시간 안에 liveliness를 알리지 않으면 <strong>liveliness 상실</strong>로 간주(장애 징후일 수 있음).</td>
</tr>
</tbody>
</table>
<p>Duration이 아닌 정책에는 <strong>system default</strong>(미들웨어 기본값) 옵션이 있고, Duration 정책에는 <strong>default</strong>(미지정, 보통 무한대로 해석)가 있다.</p>
<h2 id="qos-프로파일profiles"><a class="heading-link" aria-hidden="true" href="#qos-프로파일profiles">QoS 프로파일(Profiles)</a></h2>
<p>프로파일은 특정 용도에 맞게 잘 맞춰진 정책 묶음이다.</p>
<ul>
<li><strong>퍼블리셔·서브스크립션 default</strong>: ROS 1과 비슷한 동작을 위해 history keep last(depth 10), reliable, volatile, liveliness system default. Deadline·lifespan·lease duration은 default.</li>
<li><strong>서비스</strong>: 신뢰성이 중요. <strong>Durability는 volatile</strong>이어야 하며, 그렇지 않으면 재시작한 서버가 오래된 요청을 받을 수 있다(클라이언트는 중복 응답으로 보호되지만 서버는 아님).</li>
<li><strong>센서 데이터</strong>: 적시에 최신 샘플을 받는 것이 더 중요하므로 <strong>best effort</strong>와 <strong>작은 queue</strong>를 쓰는 프로파일.</li>
<li><strong>파라미터</strong>: 서비스 기반이므로 비슷한 프로파일이지만, 요청이 유실되지 않도록 <strong>큰 queue depth</strong>를 쓴다.</li>
<li><strong>System default</strong>: 모든 정책을 RMW 구현체 기본값으로 둔다. RMW마다 기본값이 다를 수 있다.</li>
</ul>
<p>상세 정책 값은 <a href="https://docs.ros.org/en/jazzy/Concepts/Intermediate/About-Quality-of-Service-Settings.html">원문 프로파일 링크</a>를 참고한다.</p>
<h2 id="qos-호환compatibilities"><a class="heading-link" aria-hidden="true" href="#qos-호환compatibilities">QoS 호환(Compatibilities)</a></h2>
<p>아래 내용은 퍼블리셔·서브스크립션 기준이며, <strong>서비스 서버·클라이언트</strong>에도 동일하게 적용된다.</p>
<p>퍼블리셔와 서브스크립션은 <strong>각각</strong> QoS 프로파일을 가진다. <strong>Request vs Offered</strong> 모델로 호환 여부가 정해진다. 서브스크립션은 받아들일 <strong>최소 품질(requested)</strong> 를, 퍼블리셔는 제공할 <strong>최대 품질(offered)</strong> 를 낸다. <strong>요청이 제안보다 더 엄격하지 않을 때만</strong> 연결이 성립한다. 한 퍼블리셔에 여러 서브스크립션이 붙을 수 있고 요청 프로파일이 서로 달라도 되며, 다른 퍼블리셔·서브스크립션 존재 여부는 이 쌍의 호환에 영향을 주지 않는다.</p>

<h3 id="ros-1과-비교"><a class="heading-link" aria-hidden="true" href="#ros-1과-비교">ROS 1과 비교</a></h3>
<p>ROS 1에서는 같은 토픽·같은 메시지 타입이면 퍼블리셔와 서브스크라이버가 연결되었다. ROS 2에서는 <strong>요청·제공 QoS가 호환되지 않으면 연결되지 않는다</strong>는 점을 유의해야 한다.</p>
<h2 id="qos-이벤트events"><a class="heading-link" aria-hidden="true" href="#qos-이벤트events">QoS 이벤트(Events)</a></h2>
<p>일부 QoS 정책과 연관된 <strong>이벤트</strong>가 있으며, 퍼블리셔·서브스크립션에 <strong>콜백</strong>을 등록해 처리할 수 있다(토픽 메시지 콜백과 유사).</p>
<p><strong>퍼블리셔 쪽 이벤트</strong></p>
<ul>
<li><strong>Offered deadline missed</strong>: deadline QoS에 정한 간격 안에 메시지를 발행하지 못함.</li>
<li><strong>Liveliness lost</strong>: lease duration 안에 liveliness를 알리지 못함.</li>
<li><strong>Offered incompatible QoS</strong>: 같은 토픽의 서브스크립션이 요청한 QoS를 제가 제공하는 QoS로 만족할 수 없어 연결되지 않음.</li>
</ul>
<p><strong>서브스크립션 쪽 이벤트</strong></p>
<ul>
<li><strong>Requested deadline missed</strong>: deadline QoS에 정한 간격 안에 메시지를 받지 못함.</li>
<li><strong>Liveliness changed</strong>: 구독 중인 토픽의 퍼블리셔 중 하나 이상이 lease duration 안에 liveliness를 알리지 못함.</li>
<li><strong>Requested incompatible QoS</strong>: 같은 토픽의 퍼블리셔가 제공하는 QoS가 내가 요청한 QoS를 만족하지 않아 연결되지 않음.</li>
</ul>
<h2 id="matched-이벤트"><a class="heading-link" aria-hidden="true" href="#matched-이벤트">Matched 이벤트</a></h2>
<p>QoS 이벤트 외에, <strong>퍼블리셔와 서브스크립션이 연결되거나 끊어질 때</strong> <strong>Matched 이벤트</strong>가 발생한다. 퍼블리셔·서브스크립션 각각에 콜백을 달 수 있다.</p>
<ul>
<li><strong>퍼블리셔</strong>: 토픽이 같고 QoS가 호환되는 서브스크립션을 찾았을 때, 또는 이미 연결된 서브스크립션이 끊어졌을 때.</li>
<li><strong>서브스크립션</strong>: 토픽이 같고 QoS가 호환되는 퍼블리셔를 찾았을 때, 또는 이미 연결된 퍼블리셔가 끊어졌을 때.</li>
</ul>
<p>이벤트 사용 예시는 <a href="httphttps://github.com/ros2/demos/blob/jazzy/demo_nodes_cpp/src/events/matched_event_detect.cpp">rclcpp demo</a>, <a href="https://github.com/ros2/demos/blob/jazzy/demo_nodes_py/demo_nodes_py/events/matched_event_detect.py">rclpy demo</a>를 참고하면 된다.</p>

ROS 1의 <code>spin</code>보다 실행 제어가 세밀한 <strong>명시적 Executor 클래스</strong>가 제공된다(rclcpp의 <code>executor.hpp</code>, rclpy의 <code>executors.py</code>, rclc의 <code>executor.h</code>). 기본 API는 ROS 1과 비슷하다.</p>
<p>아래는 C++ 클라이언트 라이브러리 <strong>rclcpp</strong>를 기준으로 설명한다.</p>

<p>Executor의 <code>spin()</code>이 호출되면, 현재 스레드가 rcl·미들웨어 계층에 <strong>들어온 메시지·이벤트</strong>를 묻고, 해당 <strong>콜백</strong>을 호출하는 것을 노드가 종료될 때까지 반복한다. 미들웨어 QoS와 맞추기 위해, <strong>메시지는 클라이언트 라이브러리 계층에 큐로 쌓지 않고</strong> 미들웨어에 두었다가 콜백이 처리할 때 가져온다(ROS 1과의 중요한 차이). <strong>Wait set</strong>이 “큐별로 하나의 이진 플래그” 형태로 Executor에게 “메시지 있음”을 알려주며, 타이머 만료 감지에도 쓰인다.</p>
<p>Single-Threaded Executor는 <strong>컴포넌트</strong>를 실행하는 컨테이너 프로세스에서도 사용된다(명시적 main 없이 노드가 생성·실행되는 경우).</p>

<p>세 Executor 모두 <code>add_node(..)</code>로 <strong>여러 노드</strong>를 넣어 한 Executor가 돌릴 수 있다. Multi-Threaded Executor에서 실제 병렬도는 <strong>콜백 그룹</strong> 설정에 따라 달라진다.</p>
<h2 id="callback-groups-콜백-그룹"><a class="heading-link" aria-hidden="true" href="#callback-groups-콜백-그룹">Callback groups (콜백 그룹)</a></h2>
<p>ROS 2에서는 노드의 콜백을 <strong>그룹</strong>으로 묶을 수 있다. rclcpp에서는 <code>Node::create_callback_group()</code>으로, rclpy에서는 해당 콜백 그룹 타입의 생성자로 만든다. 이 <strong>콜백 그룹은 노드가 살아 있는 동안 유지</strong>되어야 하며(예: 클래스 멤버), 구독·타이머 등을 만들 때 <strong>옵션으로 지정</strong>한다. 지정하지 않은 구독·타이머 등은 <strong>기본 콜백 그룹</strong>에 들어간다.</p>
<p><strong>콜백 그룹 타입</strong>(생성 시 결정):</p>
<ul>
<li><strong>Mutually exclusive</strong>: 이 그룹 안의 콜백은 <strong>동시에 실행되지 않음</strong>.</li>
<li><strong>Reentrant</strong>: 이 그룹 안의 콜백은 <strong>동시에 실행될 수 있음</strong>.</li>
</ul>
<p><strong>서로 다른 그룹</strong>의 콜백은 항상 병렬 실행 가능하다. Multi-Threaded Executor는 스레드 풀을 써서 위 규칙에 맞게 <strong>가능한 만큼 콜백을 병렬</strong> 처리한다. 콜백 그룹을 여러 Executor에 나눠 줄당 <code>add_callback_group(..)</code>을 쓰면, OS 스케줄러로 특정 콜백(예: 제어 루프용 구독·타이머)을 다른 콜백보다 <strong>우선시</strong>할 수 있다.</p>
<h2 id="scheduling-semantics-스케줄링-의미"><a class="heading-link" aria-hidden="true" href="#scheduling-semantics-스케줄링-의미">Scheduling semantics (스케줄링 의미)</a></h2>
<p>콜백 처리 시간이 메시지·이벤트가 도착하는 주기보다 <strong>짧으면</strong> Executor는 사실상 <strong>FIFO</strong> 순서로 처리한다. <strong>어떤 콜백이 길어지면</strong> 메시지·이벤트는 스택 하단 계층의 큐에 쌓이고, <strong>wait set</strong>은 “해당 토픽에 메시지가 있는지 여부”만 Executor에 알려준다. Executor는 이 정보만으로 서비스·액션 포함 <strong>메시지를 라운드 로빈</strong>으로 처리하며, <strong>FIFO 순서는 보장하지 않는다</strong>.</p>
<p>(타이머 이벤트를 다른 메시지보다 우선하던 동작은 Eloquent에서 제거되었다.)</p>
<h2 id="outlook-한계와-대안"><a class="heading-link" aria-hidden="true" href="#outlook-한계와-대안">Outlook (한계와 대안)</a></h2>
<p>위 세 가지 rclcpp Executor는 많은 애플리케이션에 적합하지만, <strong>실시간성</strong>(잘 정의된 실행 시간, 결정성, 실행 순서 제어)이 필요한 경우에는 다음 같은 한계가 있다.</p>
<ol>
<li><strong>스케줄링 의미가 복잡·혼합</strong>되어 있어 정형적인 타이밍 분석이 어렵다.</li>
<li><strong>우선순위 역전</strong>: 높은 우선순위 콜백이 낮은 우선순위 콜백에 막힐 수 있다.</li>
<li><strong>콜백 실행 순서</strong>를 명시적으로 제어할 수 없다.</li>
<li><strong>특정 토픽에 대한 트리거</strong>를 내장으로 제어할 수 없다.</li>
</ol>
<p>Executor 자체의 CPU·메모리 오버헤드도 무시할 수 없다. Static Single-Threaded Executor가 이를 크게 줄이지만 부족한 경우가 있다.</p>
<p>이를 일부 보완하는 방향으로:</p>
<ul>
<li><strong>rclcpp WaitSet</strong>: Executor 대신 구독·타이머·서비스·액션 등에 <strong>직접 대기</strong>할 수 있다. 결정적이고 <strong>사용자 정의 처리 순서</strong>를 구현할 수 있으며, 여러 구독의 메시지를 묶어 처리하는 것도 가능하다. (<code>examples_rclcpp_wait_set</code> 패키지 참고.)</li>
<li><strong>rclc Executor</strong>: micro-ROS용 C 클라이언트 라이브러리 rclc의 Executor. 콜백 <strong>실행 순서</strong>와 <strong>트리거 조건</strong>을 세밀하게 제어할 수 있으며, LET(Logical Execution Time) 시맨틱 아이디어를 반영한다.</li>
</ul>

<p>ROS 2는 <strong>어떤 구독(subscription)이 받은 메시지</strong>에 대한 통계를 <strong>내장 측정</strong>할 수 있다. 구독별로 통계를 수집하면 시스템 성능을 파악하거나 문제 진단에 활용할 수 있다.</p>
<p>측정 항목은 <strong>수신 메시지 주기(received message period)</strong> 와 <strong>수신 메시지 나이(received message age)</strong> 이다. 각 항목마다 <strong>평균·최대·최소·표준편차·샘플 수</strong>가 제공되며, <strong>이동 윈도우(moving window)</strong> 안에서 계산된다.</p>
<h2 id="how-statistics-are-calculated-계산-방식"><a class="heading-link" aria-hidden="true" href="#how-statistics-are-calculated-계산-방식">How statistics are calculated (계산 방식)</a></h2>
<p>각 통계 집합은 <strong>libstatistics_collector</strong> 유틸리티로 <strong>상수 시간·상수 메모리</strong>로 계산된다. 구독이 새 메시지를 받을 때마다 <strong>현재 측정 윈도우</strong>에 샘플이 하나 추가된다. 평균은 <strong>이동 평균</strong>, 최대·최소·샘플 수는 샘플마다 갱신되고, 표준편차는 <strong>Welford의 온라인 알고리즘</strong>으로 구한다.</p>

<h2 id="behavior-동작"><a class="heading-link" aria-hidden="true" href="#behavior-동작">Behavior (동작)</a></h2>
<ul>
<li><strong>기본값</strong>: 토픽 통계는 <strong>꺼져 있다</strong>. 해당 노드의 <strong>구독 설정 옵션</strong>에서 이 기능을 켜면, 그 구독에 대해 주기·나이 측정이 모두 활성화된다.</li>
<li><strong>발행</strong>: 통계는 <strong>statistics_msg/msg/MetricsMessage</strong> 타입으로, <strong>설정 가능한 주기</strong>(기본 1초)에 <strong>설정 가능한 토픽</strong>(기본 <code>/statistics</code>)에 발행된다. 이 발행 주기가 <strong>샘플 수집 윈도우 주기</strong>이기도 하다.</li>
<li><strong>나이(age)</strong>: 헤더에 타임스탬프가 없으면 나이를 구할 수 없으므로, <strong>값을 채우지 않고 NaN</strong>으로 발행한다. 아예 발행하지 않는 것보다 “측정 불가”를 명시해 <strong>신호 부재</strong>로 오해하지 않게 한다.</li>
<li><strong>주기(period)</strong>: 윈도우의 <strong>첫 번째 샘플</strong>에서는 주기 값을 내지 않는다. “이전 메시지 도착 시각”이 필요하기 때문이며, 그 다음 샘플부터 주기가 계산된다.</li>
</ul>

18:T22a4,<h2 id="ros-1---nodes-vs-nodelets"><a class="heading-link" aria-hidden="true" href="#ros-1---nodes-vs-nodelets">ROS 1 - Nodes vs. Nodelets</a></h2>
<p>ROS 1에서는 코드를 <strong>노드</strong> 또는 <strong>노드렛(nodelet)</strong> 으로 작성할 수 있다. 노드는 <strong>실행 파일</strong>로 빌드되고, 노드렛은 <strong>공유 라이브러리</strong>로 빌드된 뒤 <strong>컨테이너 프로세스</strong>가 실행 시 로드한다.</p>
<h2 id="ros-2---unified-api-통합-api"><a class="heading-link" aria-hidden="true" href="#ros-2---unified-api-통합-api">ROS 2 - Unified API (통합 API)</a></h2>
<p>ROS 2에서는 노드렛과 비슷한 방식인 <strong>컴포넌트(Component)</strong> 로 작성하는 것이 권장된다. 노드와 노드렛이 <strong>서로 다른 API</strong>를 쓰던 ROS 1의 단점을 없애고, <strong>같은 API</strong>로 통일했다.</p>
<blockquote>
<p>“자기 main을 쓰는” 노드 스타일도 가능하지만, 일반적인 경우에는 권장하지 않는다.</p>
</blockquote>
<p><strong>process layout</strong>을 <strong>배포 시점</strong>에 정하면 다음과 같은 장점이 있다.</p>
<ul>
<li><strong>노드마다 별도 프로세스</strong>: 프로세스/장애 격리, 노드 단위 디버깅에 유리.</li>
<li><strong>한 프로세스에 여러 노드</strong>: 오버헤드 감소, 필요 시 <strong>프로세스 내 통신(Intra Process Communication)</strong> 으로 효율적 통신.</li>
</ul>
<p><code>ros2 launch</code>로 위 동작을 자동화할 수 있다.</p>

<h2 id="writing-a-component-컴포넌트-작성"><a class="heading-link" aria-hidden="true" href="#writing-a-component-컴포넌트-작성">Writing a Component (컴포넌트 작성)</a></h2>
<p>컴포넌트는 <strong>공유 라이브러리</strong>로만 빌드되므로 <strong>main 함수가 없다</strong>. 보통 <code>rclcpp::Node</code>를 상속하고, 스레드를 직접 제어하지 않으므로 <strong>생성자에서 긴 작업·블로킹</strong>을 하지 말고, <strong>타이머</strong>로 주기 작업을 하거나 <strong>퍼블리셔·구독·서버·클라이언트</strong>를 만든다.</p>
<p><strong>컴포넌트로 인식되려면</strong> <code>rclcpp_components</code> 패키지의 <strong>매크로</strong>로 <strong>자기 등록</strong>해야 한다. 라이브러리가 프로세스에 로드될 때 이 등록 덕분에 컴포넌트를 찾을 수 있다(진입점 역할). 그리고 <strong>인덱스에 등록</strong>해 도구에서 찾을 수 있게 해야 한다.</p>

<h2 id="using-components-컴포넌트-사용"><a class="heading-link" aria-hidden="true" href="#using-components-컴포넌트-사용">Using Components (컴포넌트 사용)</a></h2>
<p>가장 흔한 세 가지 방식:</p>
<ol>
<li><strong>일반 컨테이너 프로세스</strong>를 띄운 뒤, 컨테이너가 제공하는 <strong>load_node</strong> 서비스를 호출해 <strong>패키지명·라이브러리명</strong>으로 컴포넌트를 로드. CLI 도구로 같은 서비스를 호출해도 된다.</li>
<li><strong>컴파일 시점에 노드가 정해진</strong> 커스텀 실행 파일을 만들어 여러 노드를 한 프로세스에 넣기. 각 컴포넌트에 <strong>헤더</strong>가 필요하다.</li>
<li><strong>런치 파일</strong>에서 <code>ros2 launch</code>로 컨테이너 프로세스를 만들고, 그 안에 <strong>여러 컴포넌트</strong>를 로드.</li>
</ol>
<p>실습은 공식 <strong>Composition</strong> 튜토리얼/데모를 참고하면 된다.</p>

<p>Open Robotics는 여러 플랫폼용 <strong>사전 빌드 ROS 2 패키지</strong>를 제공하지만, 다음 같은 이유로 <strong>크로스 컴파일</strong>을 쓰는 경우가 많다.</p>
<ul>
<li><strong>개발 머신과 목표 시스템이 다를 때</strong> (예: x86 PC에서 ARM 보드용 빌드).</li>

<h2 id="how-does-it-work-동작-방식"><a class="heading-link" aria-hidden="true" href="#how-does-it-work-동작-방식">How does it work? (동작 방식)</a></h2>
<p><strong>외부 라이브러리에 의존하지 않는</strong> 단순한 소프트웨어는 <strong>네이티브 툴체인 대신 크로스 컴파일 툴체인</strong>만 쓰면 비교적 쉽게 크로스 컴파일할 수 있다.</p>
<p>다음 요소들이 있으면 <strong>복잡도</strong>가 올라간다.</p>
<ol>
<li><strong>타깃 아키텍처 지원</strong>: 빌드 대상 아키텍처를 소프트웨어가 지원해야 하고, <strong>아키텍처별 코드</strong>(예: 어셈블리)는 타깃에 맞게 분리·활성화되어야 한다.</li>
<li><strong>의존성 선행 준비</strong>: <strong>모든 의존성</strong>(라이브러리 등)이 <strong>사전 빌드</strong>되거나 <strong>크로스 컴파일</strong>된 상태로 있어야, 그걸 쓰는 타깃 소프트웨어를 크로스 컴파일할 수 있다.</li>
<li><strong>스택/빌드 도구</strong>: <strong>colcon</strong>처럼 여러 패키지를 한꺼번에 빌드할 때는, 빌드 도구가 <strong>각 패키지의 하위 빌드 시스템</strong>에 크로스 컴파일을 넘길 수 있는 <strong>방식</strong>을 제공해야 한다.</li>
</ol>

1a:T194d,<h2 id="overview"><a class="heading-link" aria-hidden="true" href="#overview">Overview</a></h2>
<p>ROS 2는 <strong>ROS 2 계산 그래프 안의 노드 간 통신</strong>을 보호할 수 있다. 디스커버리처럼 <strong>미들웨어</strong>를 통해 동작하며(해당 보안 플러그인을 지원하는 경우), <strong>추가 소프트웨어 설치 없이</strong> 설정만으로 활성화할 수 있다. 대신 <strong>그래프 참여자마다 설정 파일</strong>이 필요하다. 이 파일들로 <strong>암호화·인증</strong>을 켜고, <strong>노드 단위·전체 그래프</strong> 정책을 정의한다. 보안 동작을 켜고 끄는 <strong>마스터 온/오프 스위치</strong>도 있다.</p>
<p>신뢰 기준(trust anchor)은 ROS 유틸로 만들거나 <strong>외부 인증 기관(CA)</strong> 를 쓸 수 있다. 제공되는 기능: 통신 <strong>암호화</strong>, 참여자 <strong>인증</strong>, 데이터 <strong>무결성</strong>, <strong>도메인 전체 접근 제어</strong>.</p>
<p>보안 서비스는 노드 간 통신에 쓰이는 <strong>DDS(Data Distribution Service)</strong> 가 제공한다. DDS Security 사양에 따라 보안 플러그인이 포함되며, ROS는 이 플러그인을 이용해 <strong>정책 기반 암호화·인증·접근 제어</strong>를 한다. <strong>설정 파일</strong>과 <strong>환경 변수</strong>로 활성화한다.</p>
<h2 id="the-security-enclave-보안-엔클레이브"><a class="heading-link" aria-hidden="true" href="#the-security-enclave-보안-엔클레이브">The Security Enclave (보안 엔클레이브)</a></h2>
<p><strong>Security Enclave</strong>는 ROS 통신을 보호하기 위한 <strong>단일 정책</strong>을 담는 단위다. 한 엔클레이브가 <strong>여러 노드</strong>, <strong>전체 ROS 그래프</strong>, 또는 보호할 프로세스·기기의 <strong>조합</strong>에 정책을 적용할 수 있다. 배포 시 <strong>프로세스·사용자·기기</strong>에 유연하게 매핑할 수 있다. 자세한 설계는 <a href="https://design.ros.org/articles/ros2_security_enclaves.html">ROS 2 Security Enclaves</a> 문서 참고.</p>
<h2 id="security-files-보안-파일"><a class="heading-link" aria-hidden="true" href="#security-files-보안-파일">Security Files (보안 파일)</a></h2>
<p>DDS 사양에 따라 한 엔클레이브는 <strong>6개 파일</strong>로 정의된다. <strong>3개는 엔클레이브 신원(identity)</strong>, <strong>3개는 권한(permissions)</strong> 이다. 6개 모두 <strong>한 디렉터리</strong>에 두고, 엔클레이브 경로를 지정하지 않고 띄운 노드는 <strong>기본 루트 엔클레이브</strong>의 파일을 사용한다.</p>

<h3 id="private-keys-비밀키"><a class="heading-link" aria-hidden="true" href="#private-keys-비밀키">Private Keys (비밀키)</a></h3>
<p>Identity·Permissions 인증서에는 각각 <strong>비밀키</strong>가 있다. 새 엔클레이브를 추가할 때: <strong>Identity CA 비밀키</strong>로 CSR(Certificate Signing Request)에 서명하고, <strong>Permissions CA 비밀키</strong>로 권한 XML에 서명해 권한을 부여한다.</p>
