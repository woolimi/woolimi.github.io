4:T194d,<h2 id="overview"><a class="heading-link" aria-hidden="true" href="#overview">Overview</a></h2>
<p>ROS 2는 <strong>ROS 2 계산 그래프 안의 노드 간 통신</strong>을 보호할 수 있다. 디스커버리처럼 <strong>미들웨어</strong>를 통해 동작하며(해당 보안 플러그인을 지원하는 경우), <strong>추가 소프트웨어 설치 없이</strong> 설정만으로 활성화할 수 있다. 대신 <strong>그래프 참여자마다 설정 파일</strong>이 필요하다. 이 파일들로 <strong>암호화·인증</strong>을 켜고, <strong>노드 단위·전체 그래프</strong> 정책을 정의한다. 보안 동작을 켜고 끄는 <strong>마스터 온/오프 스위치</strong>도 있다.</p>
<p>신뢰 기준(trust anchor)은 ROS 유틸로 만들거나 <strong>외부 인증 기관(CA)</strong> 를 쓸 수 있다. 제공되는 기능: 통신 <strong>암호화</strong>, 참여자 <strong>인증</strong>, 데이터 <strong>무결성</strong>, <strong>도메인 전체 접근 제어</strong>.</p>
<p>보안 서비스는 노드 간 통신에 쓰이는 <strong>DDS(Data Distribution Service)</strong> 가 제공한다. DDS Security 사양에 따라 보안 플러그인이 포함되며, ROS는 이 플러그인을 이용해 <strong>정책 기반 암호화·인증·접근 제어</strong>를 한다. <strong>설정 파일</strong>과 <strong>환경 변수</strong>로 활성화한다.</p>
<h2 id="the-security-enclave-보안-엔클레이브"><a class="heading-link" aria-hidden="true" href="#the-security-enclave-보안-엔클레이브">The Security Enclave (보안 엔클레이브)</a></h2>
<p><strong>Security Enclave</strong>는 ROS 통신을 보호하기 위한 <strong>단일 정책</strong>을 담는 단위다. 한 엔클레이브가 <strong>여러 노드</strong>, <strong>전체 ROS 그래프</strong>, 또는 보호할 프로세스·기기의 <strong>조합</strong>에 정책을 적용할 수 있다. 배포 시 <strong>프로세스·사용자·기기</strong>에 유연하게 매핑할 수 있다. 자세한 설계는 <a href="https://design.ros.org/articles/ros2_security_enclaves.html">ROS 2 Security Enclaves</a> 문서 참고.</p>
<h2 id="security-files-보안-파일"><a class="heading-link" aria-hidden="true" href="#security-files-보안-파일">Security Files (보안 파일)</a></h2>
<p>DDS 사양에 따라 한 엔클레이브는 <strong>6개 파일</strong>로 정의된다. <strong>3개는 엔클레이브 신원(identity)</strong>, <strong>3개는 권한(permissions)</strong> 이다. 6개 모두 <strong>한 디렉터리</strong>에 두고, 엔클레이브 경로를 지정하지 않고 띄운 노드는 <strong>기본 루트 엔클레이브</strong>의 파일을 사용한다.</p>

<h3 id="private-keys-비밀키"><a class="heading-link" aria-hidden="true" href="#private-keys-비밀키">Private Keys (비밀키)</a></h3>
<p>Identity·Permissions 인증서에는 각각 <strong>비밀키</strong>가 있다. 새 엔클레이브를 추가할 때: <strong>Identity CA 비밀키</strong>로 CSR(Certificate Signing Request)에 서명하고, <strong>Permissions CA 비밀키</strong>로 권한 XML에 서명해 권한을 부여한다.</p>

12:T197f,<h2 id="overview"><a class="heading-link" aria-hidden="true" href="#overview">Overview</a></h2>
<p><strong>tf2</strong>는 <strong>좌표 프레임(Coordinate Frame) 변환 라이브러리</strong>다. 여러 좌표계를 <strong>시간에 따라 추적</strong>하고, <strong>두 프레임 사이</strong>에서 원하는 시점의 <strong>점·벡터 등을 변환</strong>할 수 있게 한다. 프레임 간 관계를 <strong>트리 구조</strong>로 유지하며 <strong>시간에 따라 버퍼링</strong>한다.</p>
<h2 id="properties-of-tf2-특성"><a class="heading-link" aria-hidden="true" href="#properties-of-tf2-특성">Properties of tf2 (특성)</a></h2>
<p>로봇 시스템에는 world, base, gripper, head 등 <strong>시간에 따라 바뀌는 3D 좌표계</strong>가 많다. tf2는 이들을 추적해 다음 같은 질의를 가능하게 한다.</p>
<ul>
<li>5초 전 head 프레임이 world 프레임 기준으로 어디 있었는가?</li>
<li>그리퍼 안 물체의 pose를 base 기준으로 하면?</li>
<li>base 프레임의 현재 pose를 map 프레임 기준으로 하면?</li>
</ul>
<p><strong>분산 시스템</strong>에서 동작한다. 한 로봇의 모든 좌표 프레임 정보가 <strong>시스템 내 어떤 컴퓨터의 ROS 2 노드</strong>에서든 접근 가능하다. 각 노드가 자체 변환 DB를 만들거나, <strong>중앙 노드</strong>가 변환 정보를 모아 저장하는 방식 모두 가능하다.</p>
<h3 id="publishing-transforms-변환-발행"><a class="heading-link" aria-hidden="true" href="#publishing-transforms-변환-발행">Publishing transforms (변환 발행)</a></h3>
<p>변환을 발행할 때는 보통 “한 프레임에서 다른 프레임으로의 변환”으로 생각한다. <strong>데이터를 그 프레임에서 표현</strong>하는 것과 <strong>프레임 자체를 변환</strong>하는 것은 서로 <strong>역관계</strong>다. <code>geometry_msgs/msg/Transform</code> 메시지에 담는 값은 <strong>프레임 표현</strong> 쪽이다. 디버깅 시 발행한 변환은, 트리에서 어느 방향으로 조회하느냐에 따라 <strong>lookup 결과의 역</strong>이 될 수 있음을 염두에 두면 된다. tf2 라이브러리가 트리 탐색 방향에 맞춰 역변환을 처리해 준다.</p>
<h3 id="position-위치"><a class="heading-link" aria-hidden="true" href="#position-위치">Position (위치)</a></h3>
<p>한 프레임에서 관측한 점을 다른 프레임으로 옮기려면 <strong>소스 프레임 → 타깃 프레임</strong> 변환을 적용한다. <code>lookupTransform</code>에서 <strong>소스</strong>가 <code>frame_id</code> A, <strong>타깃</strong>이 <code>frame_id</code> B이면, “A에서 본 것을 B에서 본 좌표로” 변환하는 변환이 나온다. 가능하면 <strong><code>transform&#x3C;T>(target_frame, ...)</code></strong> 를 쓰는 것이 좋다. 소스 <code>frame_id</code>는 데이터 타입에서 읽고, 결과의 타깃 <code>frame_id</code>를 채우며, 수식은 내부에서 처리된다.</p>
<p>예: 루트 프레임 A가 프레임 B보다 1m 아래면, A→B 변환은 양수(위로). <strong>데이터</strong>를 B에서 A로 바꿀 때는 이 변환의 <strong>역</strong>을 써야 한다(기준이 낮아지므로 높이에 더해짐). A에서 B로 변환하면 기준이 높아지므로 높이는 줄어든다.</p>
<h3 id="velocity-속도"><a class="heading-link" aria-hidden="true" href="#velocity-속도">Velocity (속도)</a></h3>
<p>속도는 <strong>세 가지 정보</strong>로 표현한다. “움직이는 프레임 대 기준 프레임”의 속도를 <strong>관측 프레임</strong>에서 본 값이다. 예: 자동차 A의 운전자가 “지면 대비 전방 1m/s”를 A에서 관측하면 (1,0,0). 같은 속도를 지면(E)에서 보면 (0,1,0)처럼 다른 벡터가 될 수 있다. tf2 변환으로 두 표현이 동일한 속도임을 맞출 수 있다. 속도는 <strong>같은 관측 프레임</strong>에서만 더하거나 빼야 하고, 비교할 때는 먼저 <strong>같은 관측 프레임</strong>으로 변환해야 한다.</p>
<h2 id="tutorials-튜토리얼"><a class="heading-link" aria-hidden="true" href="#tutorials-튜토리얼">Tutorials (튜토리얼)</a></h2>
<p>tf2 사용은 크게 <strong>변환 수신(리스너)</strong> 와 <strong>변환 발행(브로드캐스터)</strong> 두 가지다.</p>
<ul>
<li><strong>리스너</strong>: 시스템에서 발행되는 좌표 프레임을 <strong>수신·버퍼</strong>하고, 특정 프레임 간 변환을 <strong>조회</strong>한다. <a href="https://docs.ros.org/en/jazzy/Tutorials/Intermediate/Tf2/Writing-A-Tf2-Listener-Py.html">Writing a listener</a> (Python / C++) 참고.</li>
<li><strong>브로드캐스터</strong>: 로봇의 한 부분에 해당하는 <strong>좌표 프레임의 상대 pose</strong>를 시스템에 <strong>발행</strong>한다. <a href="https://docs.ros.org/en/jazzy/Tutorials/Intermediate/Tf2/Writing-A-Tf2-Broadcaster-Py.html">Writing a broadcaster</a> (Python / C++) 참고.</li>
<li><strong>정적 브로드캐스터</strong>: <strong>시간에 따라 변하지 않는</strong> 변환은 한 번만 발행하고 이력은 저장하지 않는다. 저장·조회 부담과 발행 오버헤드를 줄인다. <a href="https://docs.ros.org/en/jazzy/Tutorials/Intermediate/Tf2/Writing-A-Tf2-Static-Broadcaster-Py.html">Writing a static broadcaster</a> (Python / C++) 참고.</li>
<li><strong>프레임 추가</strong>: <a href="https://docs.ros.org/en/jazzy/Tutorials/Intermediate/Tf2/Adding-A-Frame-Py.html">Adding a frame</a> (Python / C++)에서 고정/동적 프레임을 트리에 넣는 방법을 다룬다.</li>
<li><strong>tf2와 시간</strong>: <a href="https://docs.ros.org/en/jazzy/Tutorials/Intermediate/Tf2/Using-Time-Py.html">Using time</a>, <a href="https://docs.ros.org/en/jazzy/Tutorials/Intermediate/Tf2/Time-Travel-With-Tf2-Py.html">Traveling in time</a> (C++)에서 과거/미래 시점 조회를 다룬다.</li>
</ul>
<p>튜토리얼 목록은 공식 <a href="https://docs.ros.org/en/jazzy/Tutorials/Intermediate/Tf2/Introduction-To-Tf2-Py.html">tf2 tutorials</a> 페이지 참고.</p>

<h3 id="1-bc-behavioral-cloning"><a class="heading-link" aria-hidden="true" href="#1-bc-behavioral-cloning">1. BC (Behavioral Cloning)</a></h3>
<p><strong>소개</strong></p>
<p>가장 기본적인 모방학습 방식으로, 전문가의 상태(state)와 행동(action) 데이터를 지도학습 형태로 학습</p>
<p><strong>특징</strong></p>
<ul>
<li>구현이 매우 쉽고 빠름</li>
<li>보상 설계 없이 시연 데이터만으로 학습 가능</li>
<li>단점: 작은 에러가 누적되어 covariate shift 발생 → training 분포와 execution 분포가 달라져 일반화가 약해짐</li>
</ul>
<h3 id="2-dagger-dataset-aggregation"><a class="heading-link" aria-hidden="true" href="#2-dagger-dataset-aggregation">2. DAgger (Dataset Aggregation)</a></h3>
<p><strong>소개</strong></p>
<p>BC의 covariate shift 문제를 해결하기 위한 interactive learning 기법. 에이전트가 직접 환경에서 행동한 후, 전문가에게 그 상황에서 무엇을 했을지 피드백을 받아 데이터를 점진적으로 확장</p>
<p><strong>특징</strong></p>
<ul>
<li>초기 데이터 부족 해결</li>
<li>distribution mismatch를 줄여 안정성 향상</li>
<li>전문가의 지속적인 피드백이 필요해 현실 적용 비용이 큼</li>
</ul>
<h3 id="3-diffusion-policy"><a class="heading-link" aria-hidden="true" href="#3-diffusion-policy">3. Diffusion Policy</a></h3>
<p><strong>소개</strong></p>
<p>Diffusion Model을 행동 시퀀스 예측에 적용. noisy action sequence를 만들고 점진적으로 정제(denoising)하면서 최종 policy를 생성</p>
<p><strong>특징</strong></p>
<ul>
<li>긴 시간 동안 이어지는 복잡한 행동(long-horizon sequence)에 강점</li>
<li>연속적인 움직임을 매끄럽게 만들어 manipulation 성능이 안정적</li>
<li>transformer 기반 시퀀스 예측 보다 안정성 향상</li>
</ul>
<h3 id="4-act-action-chunking-with-transformers"><a class="heading-link" aria-hidden="true" href="#4-act-action-chunking-with-transformers">4. ACT (Action Chunking with Transformers)</a></h3>
<p><strong>소개</strong></p>
<p><a href="https://arxiv.org/pdf/2304.13705">ACT (Action Chunking with Transformers)</a></p>
<p>Transformer 기반 imitation learning 기법. 긴 action sequence를 chunk 단위로 묶어 효율적으로 학습</p>
<p><strong>특징</strong></p>
<ul>
<li>저비용 하드웨어 + 짧은 시연(10분)만으로도 bimanual task(양팔 협동 조작)를 높은 성공률로 학습</li>
<li>WidowX Arm 두 대를 활용한 ALOHA 텔레오퍼레이션 시스템에서 검증</li>
<li>2024년에는 Mobile ALOHA로 확장 → 이동(base navigation)과 양팔 조작을 동시에 학습 가능</li>
<li><a href="https://www.youtube.com/watch?v=zMNumQ45pJ8">데모 영상</a></li>
</ul>
<h3 id="5-rt-2-robotics-transformer-2"><a class="heading-link" aria-hidden="true" href="#5-rt-2-robotics-transformer-2">5. RT-2 (Robotics Transformer 2)</a></h3>
<p><strong>소개</strong></p>
<p><a href="https://arxiv.org/pdf/2307.15818">RT-2 (Robotics Transformer 2)</a></p>
<p>Google DeepMind가 발표한 Vision-Language-Action (VLA) 모델. 인터넷 규모의 웹 데이터 + 로봇 시연 데이터를 동시에 학습, 자연어·시각 정보·로봇 행동을 하나의 통합된 정책으로 연결</p>
<p><strong>특징</strong></p>
<ul>
<li>생성된 행동을 토큰화(tokenized action string)하여 언어처럼 표현, transformer으로 예측</li>
<li>Zero-shot/다단계 추론(chain-of-thought reasoning) 가능, 새로운 지시문에도 대응</li>
</ul>
<h3 id="6-π-pi-zero"><a class="heading-link" aria-hidden="true" href="#6-π-pi-zero">6. π₀ (pi-zero)</a></h3>
<p><strong>소개</strong></p>
<p><a href="https://www.physicalintelligence.company/download/pi0.pdf">π₀ (pi-zero)</a></p>
<p>OpenAI가 발표한 Vision-Language-Action 모델. RT-2와 유사하게 vision + language를 입력받아 policy를 생성하지만, action을 discrete token으로 바꾸지 않고 연속적인 action vector를 직접 출력하는 접근 방식</p>
<p><strong>특징</strong></p>
<ul>
<li>언어 + 비전 입력을 받아 continuous action sequence 산출</li>
<li>정밀한 manipulation task에서 강점</li>
</ul>
<h3 id="7-openvla"><a class="heading-link" aria-hidden="true" href="#7-openvla">7. OpenVLA</a></h3>
<p><strong>소개</strong></p>
<p><a href="https://arxiv.org/pdf/2406.09246">OpenVLA</a></p>
<p>커뮤니티 오픈소스로 RT-2 계열을 재현한 프로젝트. 누구나 학습·실험할 수 있도록 구현체와 pretrained weight를 공개</p>
<ul>
<li>입력: 이미지 + 텍스트 → transformer → 출력: tokenized action sequence (액션을 토큰화)</li>
</ul>
<p><strong>특징</strong></p>
<ul>
<li>성능은 RT-2보다 낮지만, 접근성 확대에 큰 기여</li>
<li>범용 로봇 연구를 democratize 하는 역할</li>
</ul>
<h3 id="8-smolvla"><a class="heading-link" aria-hidden="true" href="#8-smolvla">8. SmolVLA</a></h3>
<p><strong>소개</strong></p>
<p><a href="https://arxiv.org/pdf/2506.01844">SmolVLA</a></p>
<p>허깅페이스(Hugging Face)가 발표한 경량화된 VLA 모델. 대규모 GPU 없이도 실험 가능하도록 소규모 파라미터(450M)로 설계</p>
<ul>
<li>입력: 이미지 + 텍스트 → 소형 transformer → 출력: action token sequence (토큰 단위로 정의된 action들이 모인 시퀀스)</li>
</ul>
<p><strong>특징</strong></p>

<li>기본 모델 및 데이터셋 또한 오픈소스(허깅페이스)로 공개</li>
